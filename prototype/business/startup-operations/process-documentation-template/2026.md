# Process Documentation Template for Startups

## Purpose

This recipe produces a lightweight process documentation system for startups — a reusable SOP template, a prioritized library of documented processes, and a review calendar — so that only stable, repeated processes get documented while avoiding the documentation debt that kills early-stage velocity. The output is a working documentation workspace where any team member can capture, find, and maintain process knowledge in under 30 minutes per SOP. [src1]

## Prerequisites
<!-- Agents: verify ALL prerequisites before executing. Missing prerequisites = failed execution. -->

- [ ] **List of recurring team activities** available from founder or operations lead — verbal inventory or rough list is sufficient
- [ ] **Current tool stack known** — which tools the team already uses (Notion, Google Workspace, Slack, etc.)
- [ ] **Notion account** (or chosen alternative) — free signup at [notion.so](https://notion.so/signup)
- [ ] **Loom account** (optional) — for video walkthroughs, free at [loom.com](https://loom.com/signup)
- [ ] **At least 3 processes** that have been executed successfully 3+ times — do not document unstable or experimental workflows

## Constraints
<!-- Hard rules. Agents: enforce throughout execution. Violating these = documentation debt or wasted effort. -->

- Only document processes repeated 3+ times successfully — documenting unstable workflows creates maintenance burden with zero payoff. [src6]
- Keep each SOP under 1 page (approximately 500 words) — long SOPs have near-zero adoption in startup teams. [src3]
- Every SOP must have one designated owner who reviews and updates it — orphaned documents rot within weeks. [src4]
- Review quarterly for critical processes, semi-annually for supporting ones — stale SOPs are worse than no SOPs because they create false confidence. [src4]
- Do not aim for 100% process coverage — document the 20% of processes that cause 80% of confusion, delays, or errors. [src2]

## Execution Flow

### Step 1: Identify Documentation Candidates

**Duration**: 30-60 minutes
**Tool**: Spreadsheet or whiteboard

Audit your team's recurring activities to find the processes worth documenting. Apply the 3x3 filter: a process qualifies if it has been done at least 3 times AND involves at least 3 steps. [src2]

Run this exercise with your team:

1. List every recurring activity each team member does weekly or monthly
2. For each activity, record: frequency (daily/weekly/monthly), number of people who do it, number of steps involved, what happens when it goes wrong
3. Score each process: `Priority = Frequency x People x Error Impact`
4. Select the top 5-10 processes — these are your documentation candidates

```
| Process Name        | Frequency | People | Steps | Error Impact | Priority | Status     |
|---------------------|-----------|--------|-------|--------------|----------|------------|
| Customer onboarding | Weekly    | 3      | 8     | High (churn) | 72       | Document   |
| Deploy to prod      | Daily     | 2      | 5     | High (outage)| 50       | Document   |
| Invoice processing  | Monthly   | 1      | 4     | Medium       | 12       | Skip (1 person) |
| Bug triage          | Daily     | 4      | 3     | Low          | 12       | Skip (< 3 steps) |
```

**Verify**: You have 5-10 processes ranked by priority, each meeting the 3x3 threshold.
**If failed**: If fewer than 3 processes qualify, the startup is too early for formal documentation — use a shared Slack channel with pinned messages instead.

### Step 2: Set Up Documentation Workspace

**Duration**: 30-90 minutes (varies by tool path)
**Tool**: Chosen documentation platform

**Path A — Google Docs:**
1. Create a shared Google Drive folder named `SOPs`
2. Create subfolders by department: `Engineering`, `Sales`, `Operations`, `Finance`
3. Create a Google Doc named `SOP Index` with a table linking to each SOP
4. Create one template document (see Step 3) and save it as a template

**Path B — Notion Free:**
1. Create a new Notion workspace or page named `Process Library`
2. Create a database with properties: `Name` (title), `Owner` (person), `Department` (select), `Status` (select: Draft/Active/Archived), `Last Reviewed` (date), `Next Review` (date), `Priority` (select: Critical/Standard/Low)
3. Create a template button within the database using the SOP template from Step 3
4. Create filtered views: `My SOPs` (filtered by owner), `Needs Review` (filtered by next review date), `By Department`

**Path C — Slite / Notion Plus:**
1. Set up workspace with channels per department
2. Configure permissions: editors for SOP owners, viewers for all team members
3. Enable AI search features for natural-language process lookup
4. Create template and onboarding guide for new documenters

**Path D — Trainual / Process Street:**
1. Create account and configure company profile
2. Set up subjects (departments) and topics (process categories)
3. Create the SOP template as a reusable workflow
4. Assign role-based access so team members see relevant SOPs

**Verify**: Workspace is accessible to all team members, template is available, and one test SOP can be created.
**If failed**: Check sharing permissions. For Notion, verify the workspace is shared with the team email domain.

### Step 3: Apply the Lightweight SOP Template

**Duration**: 20-40 minutes per process
**Tool**: Documentation platform from Step 2

Use this one-page SOP template for every process. The template enforces brevity by design — no section should exceed 3-5 bullet points. [src3] [src5]

```markdown
# [Process Name]

**Owner:** [Name]
**Last updated:** [Date]
**Next review:** [Date — quarterly for critical, semi-annual for standard]

## When to use this
[1-2 sentences: trigger condition — what event or situation means this process should be followed]

## Before you start
- [ ] [Prerequisite 1 — tool access, permissions, or input needed]
- [ ] [Prerequisite 2]

## Steps
1. [Action verb] [specific instruction] — [expected result]
2. [Action verb] [specific instruction] — [expected result]
3. [Action verb] [specific instruction] — [expected result]
4. [Action verb] [specific instruction] — [expected result]
5. [Action verb] [specific instruction] — [expected result]

## If something goes wrong
| Problem | Fix |
|---------|-----|
| [Common error 1] | [Recovery action] |
| [Common error 2] | [Recovery action] |

## Video walkthrough (optional)
[Loom link — 2-5 minute screen recording of the process being executed]
```

**Verify**: Template fits on one page, each step starts with an action verb, the "when to use this" section is clear enough for a new hire to understand.
**If failed**: If the process cannot fit in 5-8 steps, it should be split into two separate SOPs linked to each other.

### Step 4: Document Top 5 Priority Processes

**Duration**: 2-3 hours total
**Tool**: Documentation platform + Loom (optional)

Work through the top 5 processes from Step 1, documenting each using the template from Step 3. [src1]

For each process:
1. Have the process owner write the first draft (they know it best)
2. Have someone who has never done the process read it and flag unclear steps
3. Have a third person execute the process using only the document — note where they get stuck
4. Revise based on feedback — aim for a new hire being able to follow it on day one

Optional: Record a 2-5 minute Loom video of each process being executed and embed the link in the SOP. Video walkthroughs reduce confusion by 40-60% compared to text-only documentation. [src5]

**Verify**: All 5 SOPs are complete, reviewed by at least one non-expert, and stored in the workspace.
**If failed**: If the process owner cannot articulate the steps, the process may not be stable enough to document — defer it and move to the next candidate.

### Step 5: Set Review Cadence and Prevent Over-Documentation

**Duration**: 30 minutes
**Tool**: Calendar app + documentation platform

Set up the review system to keep documentation alive without creating busywork. [src4]

1. **Assign review frequency** per SOP:
   - Critical processes (customer-facing, revenue, security): quarterly review
   - Standard processes (internal ops, reporting): semi-annual review
   - Low-priority (rarely changing, simple): annual review

2. **Create calendar reminders** for each SOP owner with the review date and a link to their SOP

3. **Define review criteria** — at each review, the owner checks:
   - Are all steps still accurate? (If not, update)
   - Has the process changed since last review? (If yes, rewrite affected steps)
   - Is this SOP still needed? (If the process stopped or changed fundamentally, archive it)
   - Can any steps be removed? (Shorter is better)

4. **Set documentation guardrails** to prevent over-documentation:
   - New SOP requests must pass the 3x3 filter (3+ executions, 3+ steps)
   - Maximum SOP count target: roughly 2x the number of team members (a 10-person team should have ~20 SOPs, not 200)
   - Archive rather than delete — outdated SOPs go to an Archive folder, not into the void

**Verify**: Calendar reminders are set for all 5 SOPs, guardrail rules are documented in the SOP Index or workspace README.
**If failed**: If the team resists scheduled reviews, start with a single monthly "SOP office hours" meeting where all reviews happen together.

## Error Handling

<!-- Common failures and recovery actions. Agents need this to handle
     execution errors without human intervention. -->

| Error | Likely Cause | Recovery Action |
|-------|-------------|----------------|
| Nobody reads or follows the SOPs | Documents are too long, hard to find, or not linked to actual workflows | Shorten to under 500 words, pin links in Slack channels where work happens, embed in onboarding checklists |
| SOPs become stale within weeks | No owner assigned or no review calendar set | Assign an explicit owner per SOP and set calendar reminders (Step 5) |
| Team resists documenting processes | Perceived as bureaucracy or busywork | Start with the single highest-pain process and demonstrate time savings; do not mandate documentation for everything |
| Documentation workspace becomes cluttered | Too many SOPs created without the 3x3 filter | Apply guardrails from Step 5; archive low-value SOPs; enforce the 2x team-size cap |
| Process changes but SOP is not updated | Triggered updates not happening | Add an SOP update step to your change management process — when a tool or workflow changes, the SOP update is part of the change, not a follow-up |

## Anti-Patterns

### Wrong: Documenting everything on day one
Startups that try to document every process before they have stable, repeated workflows create a graveyard of outdated SOPs. Processes in the first 6 months change weekly — documenting them wastes time and creates false confidence when team members follow an outdated procedure. [src6]

### Correct: Wait for the 3x3 threshold
Only document processes that have been executed at least 3 times with consistent steps. If a process is still changing, capture it as a brief note or Slack pin — not a formal SOP.

### Wrong: Writing 50-page SOP manuals
Enterprise-style documentation with exhaustive detail, approval workflows, and compliance headers. Nobody in a 10-person startup reads a 50-page operations manual. [src3]

### Correct: One-page SOPs with action verbs
Every SOP fits on one page. Each step starts with a verb. If a process needs more than 8 steps, split it into two linked SOPs.

### Wrong: No ownership or review cadence
SOPs without owners become orphaned within weeks. Teams that skip quarterly reviews find that 60%+ of their documentation is inaccurate within a year. [src4]

### Correct: Explicit owners and scheduled reviews
Every SOP has one owner (not a team, one person). Reviews are on the calendar. At each review, the owner can update, confirm, or archive.

## When This Matters

Use this recipe when a startup has grown past 3-5 people and team members are repeatedly asking the same questions about how things work, when onboarding takes longer than a week for simple roles, or when processes break because knowledge lives in one person's head. Do not use this recipe if the startup is pre-product-market-fit and processes are changing daily.
