# Startup Operations Tool Stack: Selection & Setup by Team Size

## Purpose

This recipe produces a configured operations tool stack — project management, documentation, communication, and meetings — tailored to a startup's team size, technical ratio, and budget. The output is a deployed, integrated set of tools with established workflows, channel structures, and a 12-month cost projection. [src1]

## Prerequisites
<!-- Agents: verify ALL prerequisites before executing. Missing prerequisites = failed execution. -->

- [ ] **Current tool inventory** — list of all SaaS tools currently in use with monthly costs
- [ ] **Team roster** — headcount by department (engineering, product, design, ops, sales, marketing)
- [ ] **Budget approval** — monthly per-employee tool budget confirmed by founder or finance lead
- [ ] **Admin access** — ability to create workspace admin accounts on selected platforms
- [ ] **Integration requirements** — list of must-have integrations (e.g., GitHub, CRM, design tools)

## Constraints
<!-- Hard rules. Agents: enforce throughout execution. Violating these = broken output or legal risk. -->

- Tool migration mid-sprint disrupts velocity for 2-4 weeks. Commit to selections for minimum 6 months. [src1]
- Per-seat costs compound: a 25-person team at $30/user/mo across 4 tools = $36K/year. Model growth scenarios before committing.
- Free tiers have hard limits — Slack hides messages after 90 days; Linear caps at 250 issues; Notion limits block storage. [src2]
- GDPR compliance: verify data residency options if any team member is EU-based. Slack, Notion, and Linear all offer EU data residency on paid plans.
- Avoid tool overlap — each category (PM, docs, comms) should have exactly one primary tool. Overlap creates confusion about where information lives. [src3]

## Execution Flow

### Step 1: Audit Current Tools and Identify Gaps

**Duration**: 1-2 hours
**Tool**: Spreadsheet (Google Sheets or Excel)

Inventory every SaaS tool the team uses. For each tool, record: name, category (PM/docs/comms/meetings/other), monthly cost, number of active users, satisfaction rating (1-5 from team), and integration status with other tools.

```
Tool Audit Template:
| Tool Name | Category | $/mo | Active Users | Team Rating (1-5) | Integrates With | Keep/Replace/Remove |
|-----------|----------|------|-------------|-------------------|-----------------|-------------------|
| [tool]    | PM       | $X   | N           | X                 | [tools]         | [decision]        |
```

Identify gaps: which categories have no tool? Which tools have low satisfaction scores (below 3)? Which tools overlap?

**Verify**: Audit spreadsheet covers all 4 categories (PM, docs, comms, meetings) with at least one entry each
**If failed**: Survey team leads directly — ask each lead to list tools their team uses daily

### Step 2: Match Team Profile to Stack Path

**Duration**: 30 minutes
**Tool**: Decision matrix above

Using team size, budget, and technical ratio, select the appropriate path (A through E). Within each path, apply these tiebreakers:

For **project management**:
- Mostly engineers (>60%): prefer Linear (keyboard-driven, GitHub integration, cycle-based sprints) [src2]
- Balanced or mostly non-technical: prefer Asana (intuitive adoption, less training needed) or Monday.com (visual workflows)
- Budget-constrained: Trello (generous free tier, simple Kanban) [src3]

For **documentation**:
- Under 50 people: Notion (flexible databases, templates, all-in-one workspace) [src4]
- Over 50 people or heavy Jira usage: Confluence (mature permissions, Jira integration, Rovo AI included)
- Simplicity-first teams: Slite (fast, AI-powered search, minimal setup)

For **communication**:
- Default: Slack (2,600+ integrations, channels, huddles, AI summaries on paid plans) [src6]
- Microsoft ecosystem: Teams (bundled with Microsoft 365, strong for meetings)
- Budget-zero: Discord (free voice channels, adequate for small technical teams)

**Verify**: One tool selected per category, total cost within approved budget
**If failed**: If over budget, drop to the next lower path or evaluate annual billing discounts (typically 15-20% savings)

### Step 3: Configure Project Management Tool

**Duration**: 2-4 hours
**Tool**: Selected PM platform

Set up the workspace structure:

```
Workspace Setup Checklist:
1. Create workspace with company name
2. Set up teams/spaces:
   - Engineering (sprints/cycles, bug tracking, feature requests)
   - Product (roadmap, user research, specs)
   - Design (design tasks, reviews)
   - Operations (company-wide tasks, hiring, admin)
3. Configure workflow statuses:
   - Backlog → Todo → In Progress → In Review → Done
4. Create project/board templates for recurring workflows
5. Set up integrations:
   - GitHub/GitLab (auto-link PRs to issues)
   - Slack/Teams (notifications on status changes)
   - Docs tool (embed specs in tasks)
6. Import existing tasks from old tool (CSV export → import)
7. Invite team members with appropriate roles (admin/member/guest)
```

**Verify**: All team members can create, assign, and move tasks. Integration with code repo confirmed (PR creates status update).
**If failed**: Check API token permissions. Most PM tools require admin-level tokens for bidirectional GitHub sync.

### Step 4: Set Up Documentation Platform

**Duration**: 2-3 hours
**Tool**: Selected docs platform

Create foundational workspace structure:

```
Documentation Structure:
├── Company Wiki
│   ├── Mission & Values
│   ├── Team Directory
│   └── Company Policies
├── Engineering
│   ├── Architecture Decision Records
│   ├── Onboarding Guide
│   └── Runbooks
├── Product
│   ├── PRDs (Product Requirement Docs)
│   ├── User Research
│   └── Roadmap
├── Operations
│   ├── Meeting Notes (auto-organized by date)
│   ├── Process Playbooks
│   └── Vendor & Tool Documentation
└── Templates
    ├── Meeting Notes Template
    ├── PRD Template
    ├── Post-Mortem Template
    └── Decision Log Template
```

Configure permissions: engineering docs visible to all but editable by engineering team only. Company wiki editable by all. Meeting notes auto-shared with attendees.

**Verify**: Every team member can find and edit their team's docs. Search returns relevant results within 3 keywords.
**If failed**: Restructure hierarchy — overly nested structures kill discoverability. Keep depth to 3 levels maximum.

### Step 5: Establish Communication Channels

**Duration**: 1-2 hours
**Tool**: Selected communication platform

Create channel structure that scales:

```
Channel Structure:
#general          — company-wide announcements (post-only for leadership, reactions allowed)
#random           — non-work social chat
#engineering      — engineering discussions
#product          — product discussions
#design           — design feedback and reviews
#ops              — operations and admin
#standup          — daily async standups (bot-prompted)
#incidents        — production incidents (integrate with PagerDuty/Opsgenie)
#wins             — celebrate customer wins, milestones
#feedback         — customer feedback pipeline
```

Rules to establish:
- Threads required for all replies in topic channels (reduces noise)
- @channel limited to #general and #incidents
- Bot-driven daily standups (Geekbot or equivalent) replace synchronous standups for remote teams
- Channel naming convention: `team-topic` (e.g., `eng-frontend`, `product-launches`)

**Verify**: All team members in correct channels. Test notification routing (PM tool → Slack channel on task assignment).
**If failed**: Check integration OAuth scopes. Slack integrations frequently need `channels:write` and `chat:write` scopes.

### Step 6: Integrate Tools and Verify Data Flow

**Duration**: 2-3 hours
**Tool**: All selected platforms

Connect the tools into a coherent workflow:

```
Integration Map:
PM Tool ←→ Communication Tool
  - New task assigned → message in team channel
  - Task status change → thread update

PM Tool ←→ Code Repository (GitHub/GitLab)
  - PR opened → linked to issue
  - PR merged → issue auto-closed

Docs Tool ←→ PM Tool
  - Spec doc linked in task
  - Meeting action items → tasks created

Communication Tool ←→ Calendar
  - Meeting reminders in channel
  - Huddle notes auto-posted
```

Test each integration end-to-end: create a task, assign it, link a PR, merge it, verify the task closes and Slack gets notified.

**Verify**: Complete workflow test passes — task creation to completion flows through all tools without manual intervention
**If failed**: Check webhook URLs and API tokens. Most failures are expired tokens or misconfigured webhook endpoints.

### Step 7: Set Spending Limits and Review Cadence

**Duration**: 30 minutes
**Tool**: Spreadsheet + calendar

Create the 12-month cost projection:

```
Cost Projection Template:
| Tool    | $/user/mo | Users Now | Cost Now | Users +6mo | Cost +6mo | Users +12mo | Cost +12mo |
|---------|-----------|-----------|----------|------------|-----------|-------------|------------|
| PM      | $X        | N         | $Y       | N+M        | $Z        | N+M+K       | $W         |
| Docs    | $X        | N         | $Y       | N+M        | $Z        | N+M+K       | $W         |
| Comms   | $X        | N         | $Y       | N+M        | $Z        | N+M+K       | $W         |
| Meets   | $X        | N         | $Y       | N+M        | $Z        | N+M+K       | $W         |
| TOTAL   |           |           | $Y       |            | $Z        |             | $W         |
```

Set a quarterly tool review cadence: every 3 months, survey team satisfaction (1-5 per tool), compare actual vs projected spend, and evaluate whether to upgrade, downgrade, or switch. [src5]

**Verify**: Cost projection completed for 12 months. Calendar reminder set for first quarterly review.
**If failed**: If unable to project headcount, use 2x current team for 12-month scenario (common seed-to-A growth rate).

## Quality Benchmarks

| Quality Metric | Minimum Acceptable | Good | Excellent |
|---------------|-------------------|------|-----------|
| Team adoption rate (active daily users / total users) | > 60% after 2 weeks | > 80% | > 95% |
| Tool satisfaction score (1-5 survey) | > 3.0 average | > 3.8 | > 4.5 |
| Integration reliability (successful syncs / total) | > 90% | > 97% | > 99.5% |
| Per-employee monthly tool cost vs budget | Within 120% | Within 100% | Under 80% |
| Information findability (can locate doc in < 3 searches) | > 70% success | > 85% | > 95% |

**If below minimum**: If adoption is under 60% after 2 weeks, run a 30-minute training session per team. If still low, the tool choice may not match team workflow — consider switching before sunk cost grows.

## Error Handling

| Error | Likely Cause | Recovery Action |
|-------|-------------|----------------|
| Team ignores new PM tool, tracks tasks in Slack | Old habits, insufficient onboarding | Run a 1-week forced adoption sprint: all task creation must go through PM tool. Appoint a champion per team. |
| Docs become stale within 1 month | No ownership model, no review cadence | Assign doc owners per section, set monthly freshness review reminders in calendar |
| Slack channel sprawl (50+ channels, 10-person team) | No naming convention, no archival policy | Archive channels with < 1 message/week, enforce naming convention, require approval for new channels |
| Integration breaks silently | Expired OAuth tokens, API changes | Set up a monthly integration health check. Use Zapier/Make status pages for monitoring. |
| Cost overrun from seat creep | No offboarding process removes tool access | Automate deprovisioning: when someone leaves HR system, trigger removal from all SaaS tools (use BetterCloud or Torii) |

## Anti-Patterns

### Wrong: Adopting the "everything app" approach
Choosing ClickUp or Monday.com because it claims to replace PM, docs, and comms in one tool. In practice, the docs module is weaker than Notion, the chat is weaker than Slack, and teams end up using the secondary tools anyway — now paying for both. [src3]

### Correct: Best-of-breed per category with strong integrations
Select the best tool per category (PM, docs, comms) and connect them via native integrations. A Linear + Notion + Slack stack with proper integrations outperforms a single tool trying to do everything.

### Wrong: Copying a FAANG company's tool stack at 15 people
Adopting Jira + Confluence + Teams because "Google uses it" leads to over-engineered workflows, unnecessary admin overhead, and tools designed for 10,000-person orgs squeezing a 15-person team.

### Correct: Match tools to current team size with a migration trigger
Use lightweight tools now (Linear, Notion, Slack) and define a concrete trigger for migration: "When we hit 50 engineers, evaluate Jira. When we hit 100 people, evaluate Confluence." Write the trigger down and revisit quarterly.

### Wrong: No spending controls on tool proliferation
Every team lead signs up for their own preferred tool. Within 6 months, the company has 3 PM tools, 2 docs platforms, and zero single source of truth.

### Correct: Single-tool policy per category with quarterly review
One PM tool, one docs platform, one comms tool. Exceptions require ops lead approval with a written justification. Review tool satisfaction quarterly and switch only at natural breakpoints (end of quarter, after a major milestone). [src5]

## When This Matters

Use when a startup needs to select or rationalize its operations tool stack — typically at founding, after a funding round that doubles the team, or when tool sprawl causes information fragmentation. Requires team roster and budget input. Not for enterprise tool selection (50+ people should evaluate enterprise-grade alternatives).
