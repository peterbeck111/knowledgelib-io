# Record-to-Report Integration: GL Consolidation & Intercompany Elimination

## TL;DR

- **Bottom line**: R2R integration automates the financial close cycle — extracting GL trial balances from source ERPs, mapping to a consolidated chart of accounts, translating currencies, matching intercompany transactions, and posting elimination journal entries so consolidated financials reflect only third-party activity.
- **Key limit**: Intercompany elimination requires matched pairs on both sides — one-sided entries cause imbalances that block the close. AI-powered matching achieves 80-90% auto-match rates but 10-20% still need manual resolution. [src5]
- **Watch out for**: Currency translation MUST run before intercompany elimination. Running in wrong order produces incorrect cumulative translation adjustment (CTA) entries that cascade through retained earnings. [src1]
- **Best for**: Multi-entity organizations with 3+ legal entities needing monthly/quarterly consolidated financials across one or more ERP systems.
- **Authentication**: System-to-system (OAuth 2.0 for Oracle FCCS, certificate-based for SAP RFC, Token-Based Authentication for NetSuite SuiteTalk). [src2, src3]

## System Profile

This integration playbook covers the Record-to-Report (R2R) process across the three most common enterprise consolidation platforms. R2R encompasses six sequential phases: data collection, data preparation (currency translation, net income calculation), intercompany matching and elimination, investment elimination, consolidated reporting, and balance carryforward. [src1] This card focuses on phases 1-4 — the integration-heavy steps where ERP APIs and middleware are critical.

For single-ERP environments (all entities on the same platform), native consolidation features handle most R2R tasks. Multi-ERP environments require middleware (iPaaS) or a dedicated consolidation hub to normalize data before elimination.

| System | Role | API Surface | Direction |
|---|---|---|---|
| SAP S/4HANA Group Reporting | Consolidation engine + source GL | OData v4 (CDS views), ABAP RFC | Bidirectional |
| Oracle FCCS (EPM Cloud) | Consolidation hub | REST API, EPM Automate CLI, Groovy rules | Inbound (data load) + Outbound (reports) |
| NetSuite OneWorld | Source GL + native consolidation | SuiteTalk REST/SOAP, SuiteScript 2.x | Bidirectional |
| iPaaS Middleware | Orchestration layer | Various (MuleSoft, Boomi, Workato) | Orchestrator |

### Authentication Gotchas

- SAP RFC connections require explicit authorization objects (S_RFC) — missing auth causes silent failures where the connection succeeds but data extraction returns empty results. [src1]
- Oracle FCCS REST tokens are scoped to a specific service instance — multi-environment setups (dev/test/prod) need separate credential stores. [src2]
- NetSuite TBA tokens do not expire but are revoked if the integration record or user role is modified — monitor for 401 errors on long-running integrations. [src3]

## Constraints
<!-- Agents: read this section before recommending any integration approach from this unit.
     These are hard constraints that must not be violated. -->

- **Elimination sequence is strict**: Currency translation -> intercompany matching -> intercompany elimination -> investment elimination. Violating order produces incorrect CTA and retained earnings. [src1]
- **Matched pairs required**: Both the selling entity and buying entity must record the intercompany transaction for auto-elimination. One-sided entries require manual reclassification. [src1, src3]
- **SAP ACDOCU vs ACDOCA**: Consolidation postings (ACDOCU) are separate from operational journal entries (ACDOCA). Different CDS views and APIs are needed for each. [src1]
- **Oracle FCCS cannot execute consolidation via REST API alone**: You must use EPM Automate CLI or Groovy business rules to trigger consolidation runs. REST API handles data load and journal operations only. [src2, src7]
- **NetSuite elimination subsidiaries must exist**: Each consolidation level needs a dedicated elimination subsidiary. Missing elimination subs cause the system to silently skip eliminations with no error. [src3]
- **Chart of accounts mapping is prerequisite**: Multi-ERP consolidation requires a unified financial statement item structure. Source GL accounts must map to consolidated FS items before any elimination logic runs. [src4]
- **Period must be open**: Elimination journal entries cannot post to closed periods. Late adjustments require reopening the period and re-running the entire elimination sequence.

## Step-by-Step Integration Guide

### 1. Extract Trial Balances from Source ERPs

Each source ERP provides period-end GL balances. The extraction must capture: entity code, GL account, period, functional currency amount, and intercompany partner (if tagged). [src4]

**SAP S/4HANA** — Use CDS view `I_GLAccountLineItem` or `C_TrialBalance`:
```bash
# SAP OData v4 — extract trial balance for entity 1000, period 2026-001
curl -X GET \
  "https://{sap-host}/sap/opu/odata4/sap/api_trialbalance/srvd_a2x/sap/trialbalance/0001/TrialBalance?\
$filter=CompanyCode eq '1000' and FiscalYear eq '2026' and FiscalPeriod eq '001'" \
  -H "Authorization: Bearer {token}" \
  -H "Accept: application/json"
```

**Oracle FCCS** — Load data via REST API data import:
```bash
# Oracle FCCS — import trial balance data from CSV
curl -X POST \
  "https://{epm-host}/HyperionPlanning/rest/v3/applications/{app}/jobs" \
  -H "Authorization: Bearer {token}" \
  -H "Content-Type: application/json" \
  -d '{
    "jobType": "IMPORT_DATA",
    "jobName": "Load_TB_Entity1000_202601",
    "parameters": {
      "importFileName": "TB_Entity1000_202601.csv",
      "dataLoad": true
    }
  }'
```

**NetSuite** — Use SuiteTalk REST to query GL balances:
```bash
# NetSuite REST — search trial balance by subsidiary and period
curl -X GET \
  "https://{account-id}.suitetalk.api.netsuite.com/services/rest/query/v1/suiteql" \
  -H "Authorization: OAuth {tba-header}" \
  -H "Content-Type: application/json" \
  -H "Prefer: transient" \
  -d '{"q": "SELECT account, subsidiary, SUM(amount) as balance FROM transactionLine WHERE subsidiary = 1 AND postingPeriod = 302 GROUP BY account, subsidiary"}'
```

**Verify**: Trial balance totals (debits = credits) per entity. Non-zero difference indicates extraction error.

### 2. Map Source Accounts to Consolidated Chart of Accounts

Source GL accounts from each ERP must map to a unified financial statement item structure. This is typically a one-time setup with periodic maintenance. [src4]

```python
# Input:  Source trial balance with local GL accounts
# Output: Mapped trial balance with consolidated FS items

ACCOUNT_MAPPING = {
    # SAP source accounts -> Consolidated FS items
    "SAP:0011000000": {"fs_item": "121100", "description": "Trade Receivables"},
    "SAP:0040000000": {"fs_item": "311000", "description": "Revenue - Third Party"},
    "SAP:0040100000": {"fs_item": "311100", "description": "Revenue - Intercompany"},
    # NetSuite source accounts -> Consolidated FS items
    "NS:120":         {"fs_item": "121100", "description": "Trade Receivables"},
    "NS:400":         {"fs_item": "311000", "description": "Revenue - Third Party"},
    "NS:401":         {"fs_item": "311100", "description": "Revenue - Intercompany"},
}

def map_trial_balance(source_tb, source_system):
    mapped = []
    unmapped = []
    for line in source_tb:
        key = f"{source_system}:{line['account']}"
        if key in ACCOUNT_MAPPING:
            mapped.append({
                **line,
                "fs_item": ACCOUNT_MAPPING[key]["fs_item"],
                "description": ACCOUNT_MAPPING[key]["description"]
            })
        else:
            unmapped.append(line)
    if unmapped:
        raise ValueError(
            f"{len(unmapped)} unmapped accounts — block close until resolved"
        )
    return mapped
```

**Verify**: Zero unmapped accounts. Any unmapped account blocks the close to prevent consolidation errors.

### 3. Execute Currency Translation

All entity trial balances must be translated to the group reporting currency before intercompany elimination. [src1]

```
Translation rules (IFRS/US GAAP standard):
- Balance sheet accounts:     closing rate (period-end spot rate)
- Income statement accounts:  average rate (period weighted average)
- Equity accounts:            historical rate (rate at acquisition/transaction date)
- CTA (cumulative translation adjustment): plug to balance sheet equity
```

**Verify**: Translated trial balance still balances (debits = credits). CTA account absorbs the translation difference.

### 4. Match Intercompany Transactions

Before elimination, intercompany transactions must be matched across entity pairs. Unmatched items indicate data quality issues that must be resolved. [src5]

```python
# Input:  IC transactions from all entities (post-translation)
# Output: Matched pairs + unmatched items for manual review

def match_intercompany_transactions(ic_transactions):
    """
    Match IC transactions by entity pair + amount.
    Returns (matched_pairs, unmatched_items).
    """
    # Group by entity pair (sorted to normalize direction)
    from collections import defaultdict
    pair_buckets = defaultdict(list)

    for txn in ic_transactions:
        pair_key = tuple(sorted([txn["entity"], txn["partner_entity"]]))
        pair_buckets[pair_key].append(txn)

    matched_pairs = []
    unmatched = []

    for pair_key, txns in pair_buckets.items():
        # Separate debits and credits
        debits = [t for t in txns if t["amount"] > 0]
        credits = [t for t in txns if t["amount"] < 0]

        # Exact match by absolute amount
        matched_credit_ids = set()
        for debit in debits:
            match_found = False
            for credit in credits:
                if credit["id"] in matched_credit_ids:
                    continue
                # Tolerance: allow 0.5% for FX rounding
                if abs(debit["amount"] + credit["amount"]) <= abs(debit["amount"]) * 0.005:
                    matched_pairs.append({
                        "debit_txn": debit,
                        "credit_txn": credit,
                        "variance": debit["amount"] + credit["amount"]
                    })
                    matched_credit_ids.add(credit["id"])
                    match_found = True
                    break
            if not match_found:
                unmatched.append(debit)

        # Remaining unmatched credits
        for credit in credits:
            if credit["id"] not in matched_credit_ids:
                unmatched.append(credit)

    return matched_pairs, unmatched
```

**Verify**: Unmatched items < 5% of total IC volume. >5% indicates systemic data quality issue.

### 5. Post Intercompany Elimination Journal Entries

For each matched pair, post an elimination journal entry that reverses the intercompany activity on the consolidated books. [src1, src3]

**Elimination entry pattern (revenue/expense)**:
```
Dr  Intercompany Revenue (311100)    $100,000   [eliminates seller's IC revenue]
  Cr  Intercompany COGS (411100)       $100,000   [eliminates buyer's IC expense]
Posted to: Elimination Entity (not an operating entity)
```

**Elimination entry pattern (AR/AP balance)**:
```
Dr  Intercompany Payable (211100)    $50,000    [eliminates buyer's IC payable]
  Cr  Intercompany Receivable (121100)  $50,000    [eliminates seller's IC receivable]
Posted to: Elimination Entity
```

**NetSuite** — Use SuiteScript to trigger native elimination:
```javascript
// NetSuite SuiteScript 2.x — trigger IC elimination via Period Close
// Input:  Period internal ID, subsidiary hierarchy
// Output: Elimination journal entries posted to elimination subsidiary

require(['N/task'], function(task) {
    var elimTask = task.create({
        taskType: task.TaskType.CONSOLIDATION,
        params: {
            subsidiary: 1,  // parent subsidiary
            period: 302,    // accounting period internal ID
            eliminateIntercompany: true
        }
    });
    var taskId = elimTask.submit();
    // Poll for completion
    var status = task.checkStatus({ taskId: taskId });
    log.debug('Elimination status', status.status);
});
```

**Verify**: All intercompany accounts net to zero on consolidated trial balance. Non-zero IC balance = missed elimination.

| SAP: TradingPartner (VBUND) | Partner Consolidation Unit | String | Direct map if same SAP system | Cross-system partners need explicit mapping table |
| NetSuite: subsidiary (internal ID) | Consolidation Unit ID | Integer->String | Lookup mapping table | NetSuite uses internal IDs, not entity codes |
| NetSuite: account (internal ID) | FS Item | Integer->String | Lookup mapping table | Account numbering varies by subsidiary in OneWorld |
| Oracle FCCS: Entity dimension member | Consolidation Unit | String | Direct (if FCCS is hub) | Entity names must match across data sources |
| Oracle FCCS: Account dimension member | FS Item | String | Direct (if FCCS is hub) | Account hierarchy drives elimination rules |

### Data Type Gotchas

- SAP stores amounts in the smallest currency unit for some currencies (JPY = full yen) but in decimal for others (USD = dollars.cents). Always check the CURRDEC currency decimal table before transformation. [src1]
- NetSuite intercompany transactions use the subsidiary's base currency. When entities have different base currencies, the elimination entry includes a CTA line for the translation difference. [src3]
- Oracle FCCS expects data load files in a specific "point of view" format: Entity, Account, Period, Year, Scenario, Currency, Data — column order matters. Wrong column order causes silent data load failures. [src2]
- Intercompany partner identification differs by system: SAP uses Trading Partner (VBUND field), NetSuite uses IC Customer/Vendor records, Oracle FCCS uses a dedicated Intercompany dimension. Cross-system matching requires a partner mapping table. [src4]

## Error Handling & Failure Points

### Common Error Codes

| Code | System | Meaning | Cause | Resolution |
|---|---|---|---|---|
| ACDOCU_POST_ERROR | SAP | Consolidation posting failed | Missing posting level config or locked period | Verify posting level 20 is configured; check period lock status |
| EPMCSS-00010 | Oracle FCCS | Data import validation error | Column mismatch or invalid dimension member | Validate data file against FCCS dimension members before load |
| IC_ELIM_IMBALANCE | NetSuite | Elimination entry out of balance | One-sided IC transaction or missing elimination account | Run IC Reconciliation Report; fix source transactions [src3] |
| INVALID_PERIOD | All systems | Cannot post to period | Period is closed or not yet opened | Reopen period or adjust posting date |
| NO_ELIM_SUB | NetSuite | No elimination subsidiary found | Elimination subsidiary not configured for consolidation level | Create elimination subsidiary under correct parent [src3] |
| FX_RATE_MISSING | All systems | Exchange rate not available | Rate table incomplete for entity's currency | Load exchange rates before running translation |

### Failure Points in Production

- **One-sided intercompany transactions**: Entity A records IC sale but Entity B misses the corresponding IC purchase. Elimination runs but only catches one half, causing an imbalanced consolidated trial balance. Fix: `Run IC Reconciliation Report before elimination. Block close if unmatched items exceed threshold.` [src3]
- **Late adjustments after elimination**: Accountant posts a journal entry to an already-eliminated period. The elimination journals are now stale. Fix: `Re-run the entire elimination sequence after any post-elimination adjustment. In NetSuite, the system allows multiple elimination runs per period.` [src3]
- **Currency rate timing mismatch**: Entity A uses month-end closing rate from Reuters at 4pm, Entity B uses Bloomberg rate at 6pm. Small FX differences accumulate across many transactions. Fix: `Centralize exchange rate source. Load rates to consolidation hub first, then push to all source ERPs.` [src1]
- **Chart of accounts drift**: A subsidiary adds new GL accounts mid-year without updating the consolidation mapping table. New accounts are silently excluded from consolidated reporting. Fix: `Automated unmapped account detection in data validation step. Zero-tolerance policy — any unmapped account blocks the close.` [src4]
- **Elimination subsidiary access misconfiguration in NetSuite**: The elimination subsidiary does not have access to all intercompany accounts. Elimination entries fail silently. Fix: `Verify elimination subsidiary is listed on each intercompany account's subsidiary access list.` [src3]
- **SAP posting level conflicts**: Consolidation adjustments at posting level 20 conflict with manual top-side adjustments at the same level. Fix: `Reserve distinct posting levels for automated eliminations (20), manual adjustments (30), and audit entries (40).` [src1]

## Anti-Patterns

### Wrong: Running elimination before completing currency translation

```
// BAD — Elimination uses untranslated amounts
Step 1: Extract trial balances (local currency)
Step 2: Run intercompany elimination ← amounts in different currencies!
Step 3: Currency translation
Step 4: Consolidated reporting ← CTA entries incorrect
```

### Correct: Enforce strict R2R phase sequence

```
// GOOD — Translation before elimination ensures amounts are comparable
Step 1: Extract trial balances (local currency)
Step 2: Currency translation → all amounts in group currency
Step 3: Run intercompany elimination ← amounts now comparable
Step 4: Consolidated reporting ← CTA entries correct
```

### Wrong: Using generic customers/vendors for intercompany transactions

```
// BAD — NetSuite cannot auto-identify IC transactions
// Subsidiary A creates invoice to "Subsidiary B" using a regular customer record
// Result: Transaction not flagged as intercompany, elimination skips it
```

### Correct: Use designated intercompany entity records

```
// GOOD — NetSuite auto-flags and pairs IC transactions
// Subsidiary A creates invoice using IC-SubsidiaryB (intercompany customer)
// System automatically: flags for elimination, pairs with corresponding AP,
//   populates "Due To/From" fields, includes in IC Reconciliation Report
```
[src3]

### Wrong: Building consolidation logic in middleware instead of the consolidation platform

```
// BAD — iPaaS calculates elimination entries
// MuleSoft flow: extract TB → calculate eliminations → post net JEs to ERP
// Problems: no audit trail, no CTA handling, no investment elimination,
//   breaks when ownership % changes, no support for partial/proportional consolidation
```

### Correct: Use middleware for data movement only; consolidation logic in dedicated platform

```
// GOOD — iPaaS extracts and loads; consolidation platform does the accounting
// MuleSoft flow: extract TB from each ERP → map accounts → load to FCCS
// FCCS: currency translation → IC matching → elimination → investment elim → reporting
// Result: full audit trail, proper CTA, minority interest, ownership % changes handled
```
[src4]

## Common Pitfalls

- **Pitfall: Assuming single-ERP eliminates multi-ERP complexity**: Even if 80% of entities are on SAP, the remaining 20% on other systems require the same consolidation rigor. Fix: `Design for multi-ERP from day one. Use a consolidation hub even if most entities are on one platform.` [src4]
- **Pitfall: Not testing with realistic intercompany volumes**: Dev/test environments often have 10 IC transactions; production has 10,000. Matching algorithms that work at small scale may timeout or produce false matches at volume. Fix: `Load-test with production-volume IC data. Set matching timeout thresholds.` [src5]
- **Pitfall: Ignoring intercompany profit in inventory**: Revenue/expense and AR/AP elimination is straightforward, but unrealized profit on IC-sourced inventory still sitting on the buyer's balance sheet is frequently missed. Fix: `Track IC margin on inventory items. Post unrealized profit elimination for IC goods not yet sold to third parties.` [src1]
- **Pitfall: Manual elimination entries alongside automated ones**: Accountants post manual elimination JEs to "fix" issues while the automated process also runs, causing double-elimination. Fix: `Disable manual JE posting to elimination entities during automated close. Use clear posting level separation.` [src3]
- **Pitfall: Period-end race condition**: Entity A closes and submits data on Day 3, Entity B submits on Day 5, but the consolidation team starts elimination on Day 4 with incomplete data. Fix: `Implement a close readiness gate — all entities must submit and validate before elimination runs.` [src4]
- **Pitfall: Treating CTA-Elimination as a rounding account**: CTA-E entries should reflect real translation differences on IC items. Large or unexpected CTA-E balances indicate configuration errors, not rounding. Fix: `Monitor CTA-E monthly. Investigate any balance exceeding 1% of total IC activity.` [src3]

## When to Use / When Not to Use

| Use When | Don't Use When | Use Instead |
|---|---|---|
| Multi-entity organization needs consolidated financials (IFRS/GAAP) | Single legal entity with no subsidiaries | Direct GL reporting from ERP |
| Intercompany transactions exist between entities | No intercompany activity between entities | Standard financial reporting |
| Monthly/quarterly close requires automated elimination | Annual close with <10 IC transactions (manual is acceptable) | Manual elimination JEs |
| Multi-ERP environment needs unified consolidation | All entities on same ERP with native consolidation working | Native ERP consolidation features |
| Statutory reporting requires full audit trail of eliminations | Management reporting only, no audit requirements | Data warehouse consolidation |
| Organization has >5 legal entities with cross-entity activity | 2-3 entities with minimal IC activity | Simplified IC elimination in single ERP |

## Important Caveats

- Consolidation platforms evolve rapidly — SAP Group Reporting, Oracle FCCS, and NetSuite all release new features quarterly. Verify specific API endpoints and configuration steps against current vendor documentation before implementation.
- This card covers the three most common platforms. Organizations using Workday, Microsoft Dynamics 365 F&O, or Infor require different integration patterns not detailed here.
- Intercompany profit elimination on inventory and fixed assets is significantly more complex than revenue/expense or AR/AP elimination. It requires tracking of IC margin, inventory aging, and asset depreciation schedules — typically a phase 2 implementation.
- IFRS 10 and US GAAP ASC 810 have different rules for consolidation scope (who to consolidate), variable interest entities, and non-controlling interests. This card covers the integration mechanics, not the accounting policy decisions.
- Regulatory requirements for statutory consolidation vary by jurisdiction. EU entities may need ESEF/iXBRL tagging; US public companies need SEC XBRL. The R2R integration must accommodate these downstream reporting requirements.
