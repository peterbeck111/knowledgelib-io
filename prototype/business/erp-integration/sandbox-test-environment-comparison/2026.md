# ERP Sandbox & Test Environment Comparison (2026)

## TL;DR

- **Bottom line**: Sandbox capabilities vary dramatically across ERPs — Salesforce offers the most granular self-service sandbox tiers, while SAP S/4HANA Cloud Public Edition is the most restrictive with no traditional client copy support. Budget 1-5 days for provisioning on most platforms.
- **Key limit**: Refresh cycles range from daily (Salesforce Developer) to once per 29 days (Salesforce Full) to weekly forced overwrites (Workday). Plan testing windows around these constraints.
- **Watch out for**: API limits in sandboxes are typically lower than production across all ERPs. Integration tests that pass in sandbox may fail in production due to different throttling, or vice versa — sandbox may not expose production-scale concurrency issues.
- **Best for**: Evaluating which ERP sandbox model fits your CI/CD pipeline, release validation cadence, and integration testing requirements.
- **Cost factor**: Most ERPs include 1-3 sandbox environments with subscription; additional sandboxes range from free (Acumatica via partner) to $10K+/year (Salesforce Full Copy add-on, SAP optional sandbox tenant).

## System Profile

This comparison card covers sandbox and test environment capabilities across nine major ERP platforms as of early 2026. It focuses on cloud deployments, as sandbox provisioning for on-premise installations is fundamentally different (customer-managed infrastructure). Each ERP takes a different philosophical approach: some offer self-service sandbox management (Salesforce, Business Central), others require vendor engagement (SAP, IFS), and some blend both approaches (Oracle, NetSuite).

| System | Vendor | Deployment | Primary Sandbox Model |
|---|---|---|---|
| Salesforce | Salesforce | Cloud | 4-tier self-service (Developer, Developer Pro, Partial Copy, Full) |
| SAP S/4HANA Cloud | SAP | Cloud | Starter + optional Sandbox tenant (vendor-managed) |
| Oracle ERP Cloud | Oracle | Cloud | Sandboxes within Fusion Applications + test instances |
| NetSuite | Oracle | Cloud | Sandbox accounts (refresh from production/other sandbox) |
| Dynamics 365 F&O | Microsoft | Cloud | Tier-1 through Tier-5 via LCS (Lifecycle Services) |
| Business Central | Microsoft | Cloud | Up to 3 sandboxes per tenant (self-service) |
| Workday | Workday | Cloud | Sandbox + Sandbox Preview + Implementation tenants |
| IFS Cloud | IFS | Cloud | Pre-production + sandbox (IFS-managed) |
| Acumatica | Acumatica | Cloud | Sandbox instance via partner (snapshot/restore model) |

## Detailed Sandbox Profiles

### Salesforce

Salesforce offers the most granular sandbox model with four distinct tiers. [src1]

| Sandbox Type | Storage | Data Included | Refresh Interval | Best For |
|---|---|---|---|---|
| Developer | 200 MB | Metadata only | 1 day | Individual developer work, unit testing |
| Developer Pro | 1 GB | Metadata only | 1 day | Team development, integration testing |
| Partial Copy | 5 GB | Metadata + sample data (10K records/object via template) | 5 days | QA testing, staging with realistic data |
| Full | Matches production | Full production data copy | 29 days | UAT, performance testing, data migration dry runs |

**Key details**:
- Sandboxes inactive for 180 days are deleted after 90/120/150-day warnings
- Developer and Developer Pro sandboxes have lower API call limits (15,000/24h for Developer vs 100,000/24h for Enterprise production)
- Sandbox preview instances get major release updates before production, enabling pre-release testing
- Refresh copies metadata; Full Copy additionally copies all data records
- Partial Copy uses sandbox templates to control which objects and how many records are copied

### SAP S/4HANA Cloud Public Edition

SAP's cloud-native approach fundamentally differs from traditional on-premise SAP landscape management. Client copies and remote client copies — standard practice in SAP ECC — are not supported in the Public Edition due to multi-tenancy and SaaS architecture. [src2]

| Environment Type | Purpose | Provisioning | Refresh |
|---|---|---|---|
| Starter System | Initial Fit-to-Standard workshops | Included with subscription; provisioned by SAP | Decommissioned after go-live |
| Quality System | Testing, UAT | Part of standard 3-system landscape | SAP Test Data Refresh Service (limited) |
| Sandbox Tenant | Evaluate configurations and extensions | Optional add-on (additional subscription fee) | On-demand via SAP (not self-service) |

**Key details**:
- 3-system landscape: Development → Quality → Production (standard); optional parallel line for sandbox
- No traditional client copy or transport between sandbox and main line
- SAP BTP (Business Technology Platform) provides additional development and testing capabilities
- Private Edition offers more flexibility with system copies similar to traditional on-premise

### Oracle ERP Cloud (Fusion Applications)

Oracle uses a dual-layer sandbox model: lightweight configuration sandboxes within a Fusion instance, and full environment copies for comprehensive testing. [src3]

| Environment Type | Purpose | Data Included | Refresh |
|---|---|---|---|
| Sandbox (in-instance) | Configuration testing, UI customization | Shares production data (isolated config layer) | Publish changes; no data refresh needed |
| Test Instance | Full environment for UAT, integration testing | Full data copy from production | On-demand (source and target must match version/patch level) |
| Development Instance | Custom development | Minimal data | On-demand |

**Key details**:
- Cannot refresh a production environment — only test/dev environments can be targets
- Source and target must have matching Fusion Applications version and patch level (including monthly and exception updates)
- During refresh, target environment is unavailable; source is unaffected
- Keep the number of open publishable sandboxes low for system performance
- Make small, focused configurations in separate sandboxes rather than large changes in one

### NetSuite

NetSuite provides sandbox accounts as full copies of production with some exclusions. Available on Premium, Enterprise, and Ultimate service tiers. [src4]

| Capability | Details |
|---|---|
| Refresh source | Production account (all tiers); another sandbox (Premium, Enterprise, Ultimate only) |
| Data copied | All configurations, data, user passwords, customizations |
| Data NOT copied | Customer Center role assignments, OAuth tokens, OAuth 2.0 app authorizations |
| Activation window | 14 days to activate after refresh; expired copies are deleted |
| Refresh frequency | On-demand (no minimum interval) |
| Snapshot timing | Data reflects the Snapshot Date, not the exact moment of refresh request |

**Key details**:
- OAuth tokens and OAuth 2.0 authorizations must be recreated after every refresh
- Check the Snapshot Date field to understand what point-in-time data the sandbox contains
- SuiteCloud Development Framework (SDF) supports sandbox-to-production deployment of customizations
- Governance unit limits in sandbox match production tier

### Dynamics 365 Finance & Operations

Microsoft uses Lifecycle Services (LCS) to manage environments across multiple tiers. [src5]

| Tier | Type | Purpose | Performance | Provisioning |
|---|---|---|---|---|
| Tier-1 | Developer/Test | Development, unit testing | Single-box, lower performance | Self-service via LCS |
| Tier-2 | Standard Acceptance Testing | UAT, integration testing | Multi-box, moderate | Included with subscription |
| Tier-3 | Pre-production | Performance testing | Higher performance | Additional capacity |
| Tier-4 | Premium | Load testing | Near-production performance | Additional capacity |
| Tier-5 | Premium+ | Full production scale | Production-grade | Additional capacity |

**Key details**:
- Database refresh from production to sandbox UAT is done via LCS
- Refresh during off-peak hours recommended; limit to one refresh operation at a time
- Total database storage across all environments must not exceed 80 GB (default; expandable)
- One non-production environment included at no additional cost (with sufficient storage)
- Each additional production environment includes 3 additional sandbox environments + 4 GB tenant-wide DB capacity

### Dynamics 365 Business Central

Business Central takes a simple, self-service approach to sandboxes. [src6]

| Capability | Details |
|---|---|
| Included sandboxes | 3 per tenant (Premium and Essential) |
| Provisioning | Self-service via Admin Center (minutes) |
| Refresh method | Delete sandbox + create new from production (no in-place refresh) |
| Data copied | Full production copy (email accounts of type "Microsoft 365" and "Current User" deleted) |
| Storage | 80 GB total across all environments (default) |
| Backups | Production is backed up regularly; sandboxes are NOT backed up |

**Key details**:
- No in-place refresh — must delete and recreate to get fresh production data
- Sandbox performance runs in a lower Azure performance tier than production
- Preview AppSource apps installed in sandbox block copy-to-production — must update to Public versions first
- Good CI/CD support via AL extensions and Azure DevOps pipelines
- Can copy sandbox-to-sandbox and production-to-sandbox; sandbox-to-production is restricted

### Workday

Workday takes a unique approach with automatic weekly sandbox refreshes. [src7]

| Tenant Type | Purpose | Refresh Cycle | Data Included |
|---|---|---|---|
| Sandbox | General testing, configuration | Weekly (every Friday night) | Full production mirror (workers, positions, cost centers, journals, transactions) |
| Sandbox Preview | Pre-release testing | Twice per year (before major releases) | Full production mirror + upcoming release features |
| Implementation | Extended testing cycles | On-demand (customer-controlled) | Full production replica |

**Key details**:
- Weekly sandbox refresh overwrites ALL configuration and test data — anything not migrated to production is lost
- Can request to skip a scheduled refresh (max 2 consecutive weeks)
- Sandbox Preview receives new feature updates ahead of production
- Implementation tenant provides the most flexibility for extended testing
- No self-service provisioning — Workday manages tenant lifecycle
- API limits in sandbox match production

### IFS Cloud

IFS Cloud uses a managed environment model with semi-annual release cadence. [src8]

| Environment Type | Purpose | Provisioning | Refresh |
|---|---|---|---|
| Pre-production | Release validation, regression testing | Included with subscription | Semi-annual (aligned with R1/R2 releases) + on-demand |
| Sandbox | Impact analysis, configuration testing | Optional (IFS-managed) | On-demand via IFS Cloud Support |

**Key details**:
- Standard deployment: sandbox deployment via IFS Cloud Support (typically Week 2 of update cycle)
- IFS Managed Services handles lifecycle orchestration: environments, updates, backups, rollbacks
- Semi-annual release cadence (R1, R2) with monthly service updates for security and performance
- Sandbox-first approach recommended for impact analysis before production deployment
- No self-service sandbox provisioning — requires IFS Cloud Support engagement

### Acumatica

Acumatica offers a partner-mediated sandbox model with flexible snapshot-based data management. [src8]

| Capability | Details |
|---|---|
| Provisioning | Via Acumatica Partner; typically 1-5 business days |
| Cost | Free sandbox for testing/training prior to automated upgrades (via partner) |
| Data copy | Full company copy or partial via snapshot (company-level) |
| Refresh | On-demand via snapshot restore |
| Automation | Snapshot creation and restoration via UI or API |

**Key details**:
- Acumatica manages setup, backups, software updates, and monitoring
- Company-level snapshots allow copying entire company data or partial data sets to test environments
- Consumption-based licensing means sandbox usage counts toward compute/storage allocation
- Can install locally for development (on-premise option available alongside cloud)

## Constraints

<!-- Agents: read this section before recommending any sandbox strategy. These are hard constraints. -->

- Salesforce Full sandbox refresh is limited to once per 29 days — cannot get fresher production data more frequently without using Partial Copy (5-day cycle) [src1]
- SAP S/4HANA Cloud Public Edition does not support client copies — customers expecting traditional SAP landscape management will be disappointed [src2]
- Oracle ERP Cloud requires version/patch matching for environment refresh — you cannot refresh from a newer-patched production to an older test instance [src3]
- NetSuite OAuth tokens and app authorizations are wiped on every sandbox refresh — integration tests must re-authenticate after each refresh [src4]
- Business Central sandboxes are NOT backed up — if the sandbox corrupts, you lose all test configuration since last production copy [src6]
- Workday overwrites sandbox every Friday night — any custom configuration or test data not migrated to production is destroyed [src7]
- IFS Cloud sandbox provisioning is not self-service — budget days for environment requests through IFS Cloud Support [src8]
- Acumatica sandbox provisioning requires partner involvement — no direct self-service option [src8]

## Common Pitfalls

- **Assuming sandbox API limits match production**: Salesforce Developer sandboxes have 15,000 API calls/24h vs 100,000+ for Enterprise production. Load tests in sandbox will give misleading results. Fix: `Use Full or Partial Copy sandbox for performance testing, or explicitly account for limit differences in test plans`. [src1]
- **Not re-creating OAuth tokens after NetSuite refresh**: All OAuth tokens and app authorizations are wiped during refresh. Automated integration tests will fail silently. Fix: `Add a post-refresh hook in your CI/CD pipeline that re-creates sandbox tokens before running integration tests`. [src4]
- **Building config in Workday Sandbox without migrating**: The weekly Friday night refresh destroys all sandbox-only configuration. Fix: `Migrate completed configurations to production or implementation tenant before Friday; or request a refresh skip (max 2 consecutive weeks)`. [src7]
- **Expecting SAP client copy behavior in S/4HANA Cloud PE**: Customers from on-premise SAP backgrounds expect remote client copies. These are not available in Public Edition. Fix: `Use SAP Test Data Refresh Service or SAP BTP for development/testing needs`. [src2]
- **Ignoring Business Central sandbox backup limitations**: Sandboxes are not backed up. A corruption event means starting over from production. Fix: `Export data via Rapid Start before making major sandbox changes; treat sandbox config as disposable`. [src6]
- **Testing integrations in Oracle sandbox without version matching**: Source and target must have identical Fusion Applications version and patch level. A patch mismatch blocks refresh. Fix: `Coordinate refresh timing with Oracle's monthly update schedule; refresh immediately after both environments are patched`. [src3]
- **Not budgeting time for IFS Cloud sandbox requests**: No self-service provisioning means days of lead time. Fix: `Submit sandbox requests at least 2 weeks before testing milestones; align with IFS's standard Week 2 sandbox deployment cadence`. [src8]
- **Letting Salesforce sandboxes go inactive**: Sandboxes unused for 180 days are automatically deleted. Fix: `Log into each sandbox at least once per quarter; set calendar reminders at 60-day intervals`. [src1]

## Important Caveats

- Sandbox capabilities change with each major release — verify current limits against vendor documentation before making architecture decisions
- Additional sandbox costs are often negotiable during contract renewals; list prices may not reflect actual pricing
- "Full copy" does not always mean identical to production — each vendor excludes certain data types (tokens, email configs, certain role assignments)
- Performance in sandbox environments is typically lower than production across all vendors; never use sandbox performance metrics for capacity planning
- Multi-region deployments may have different sandbox availability and refresh constraints depending on data residency requirements
- This comparison covers cloud deployments only; on-premise and hybrid deployments have fundamentally different sandbox provisioning models
