# Tray.ai (Tray.io) Integration Capabilities for Complex ERP Orchestration

## TL;DR

- **Bottom line**: Tray.ai is a developer-oriented iPaaS with three build experiences (Tray Code, Tray Build, Tray Chat) and 700+ connectors, best suited for complex multi-system ERP orchestration where workflow logic goes beyond simple point-to-point mapping.
- **Key limit**: 6 MB data payload limit between workflow steps; 30 req/s API rate limit (1,800/min) with burst to 50 req/s; 1,000 max concurrent Call Connector requests. [src1, src2]
- **Watch out for**: No pre-built SAP ERP connector — you must use HTTP Client or build a custom connector via the CDK; task credit consumption scales with step count, not workflow count. [src5]
- **Best for**: Complex orchestration across 3+ systems with branching logic, conditional routing, data transformation, and error handling that exceeds what recipe-based tools can express.
- **Authentication**: OAuth 2.0 (Authorization Code + Client Credentials), API Key, Basic Auth, Custom Service Auth — centralized credential store with auto-refresh. [src6]

## System Profile

Tray.ai (rebranded from Tray.io in 2024) is a cloud-native iPaaS built on the Universal Automation Cloud (UAC) architecture. It targets technical teams that need full control over integration logic while also enabling business technologists through low-code and AI-assisted build experiences. Unlike MuleSoft's API-led connectivity model or Workato's recipe-centric approach, Tray takes a workflow-first approach with deep support for branching, looping, conditional logic, and parallel execution.

The platform operates across three tiers: Pro (entry-level), Team (mid-market), and Enterprise (full governance). All tiers share the same runtime engine but differ in task credits, workspace count, log retention, and governance controls. Tray does NOT offer an on-premise deployment option — it is cloud-only, hosted on AWS.

## Authentication

Tray.ai provides a centralized authentication store that manages credentials for all connected services. Workflows reference stored auth objects rather than embedding credentials directly. [src6]

| Flow | Use When | Token Lifetime | Refresh? | Notes |
|---|---|---|---|---|
| OAuth 2.0 Authorization Code | User-context operations, most SaaS apps | Per-service default | Yes (automatic) | Default and preferred flow; supports PKCE |
| OAuth 2.0 Client Credentials | Server-to-server, machine-to-machine | Per-service default | Yes (automatic) | For APIs that don't need user context |
| API Key | Simple service auth (Stripe, SendGrid, etc.) | Until revoked | N/A | Stored encrypted in Tray vault |
| Basic Auth | Legacy systems, on-premise APIs | Per-session | N/A | Username/password stored encrypted |
| Custom Service Auth | Non-standard auth flows | Variable | Configurable | Build custom auth via Connector Builder |

### Authentication Gotchas

- OAuth tokens are stored per-workspace — moving a workflow between workspaces requires re-authentication of all connected services. [src1]
- Tray automatically refreshes tokens on 401/403 responses with 3 retries (1st immediate, 2nd-3rd with ~20s delay) — if all fail, the workflow step errors. [src1]
- For embedded (white-label) deployments, OAuth redirect URLs use the format `<org_name>.integration-authentication.com/oauth2/token` — you must register this URL with each OAuth provider. [src3]
- Connector-level auth is separate from Tray platform API auth — the platform APIs use user tokens obtained via GraphQL mutations. [src2]

## Constraints
<!-- Agents: read this section before recommending any integration approach from this unit.
     These are hard constraints that must not be violated. -->

- **6 MB step payload limit**: Data passed between workflow steps cannot exceed 6 MB. For large datasets, use CSV connectors (up to 1 GB), Data Storage, or file-based intermediaries. This is the #1 constraint developers hit on data-intensive ERP integrations. [src1]
- **No native SAP connector**: SAP ERP, S/4HANA, and SAP BTP do not have pre-built connectors. You must use the HTTP Client connector with custom authentication or build a connector via the CDK. [src5]
- **Task credits are per-step, not per-workflow**: Each connector operation, data transformation, boolean, and loop iteration consumes a task credit. A 20-step workflow processing 1,000 records via a loop consumes 20,000+ credits per execution. [src7]
- **Cloud-only deployment**: No on-premise or hybrid option. All data transits through Tray's AWS infrastructure. Organizations with strict data residency requirements must verify Tray's data processing regions. [src6]
- **Webhook rate limiting**: Webhook triggers using "Validate and respond" or "Await workflow and respond" mode are rate-limited at high volume (hundreds/second triggers HTTP 429). Only "Auto respond with status 200" mode is unthrottled. [src1]
- **Log retention is plan-dependent**: Pro gets 7 days, Enterprise up to 30 days. There is no long-term log archival built in — you must export logs to external systems for compliance. [src7]
- **Generated file TTL**: Files created during workflow execution (CSVs, exports) are automatically deleted after 6 hours. Download URLs also expire at 6 hours. [src1]

## Code Examples

### Python: Trigger a Tray Workflow via Webhook

```python
# Input:  Webhook URL from Tray workflow, ERP event payload
# Output: HTTP 200 (async) or workflow result (sync)

import requests
import json

TRAY_WEBHOOK_URL = "https://YOUR_TRAY_WEBHOOK_URL"

def trigger_tray_workflow(event_type: str, payload: dict) -> dict:
    """Trigger a Tray.ai workflow via webhook with ERP event data."""
    headers = {
        "Content-Type": "application/json",
        # Optional: Add HMAC signature for webhook verification
        "X-Webhook-Signature": compute_hmac(payload)
    }

    response = requests.post(
        TRAY_WEBHOOK_URL,
        headers=headers,
        json={"event": event_type, "data": payload},
        timeout=30  # 5-min max for sync, but set reasonable client timeout
    )

    if response.status_code == 429:
        # Tray rate limit hit — back off and retry
        import time
        time.sleep(2)
        return trigger_tray_workflow(event_type, payload)

    response.raise_for_status()
    return response.json() if response.text else {"status": "accepted"}


def compute_hmac(payload: dict) -> str:
    """Compute HMAC-SHA256 for webhook verification."""
    import hmac, hashlib
    secret = b"YOUR_WEBHOOK_SECRET"
    msg = json.dumps(payload, sort_keys=True).encode()
    return hmac.new(secret, msg, hashlib.sha256).hexdigest()


# Example: Send order created event
result = trigger_tray_workflow("order_created", {
    "order_id": "ORD-12345",
    "customer_id": "CUST-001",
    "total": 2500.00,
    "currency": "USD"
})
print(f"Workflow triggered: {result}")
```

### JavaScript/Node.js: Use Tray Embedded API (GraphQL)

```javascript
// Input:  Tray user token, workflow ID
// Output: Workflow execution status and instance details

// npm install graphql-request@6.x

import { GraphQLClient, gql } from "graphql-request";

const TRAY_GRAPHQL_ENDPOINT = "https://tray.io/graphql";
const USER_TOKEN = process.env.TRAY_USER_TOKEN;

const client = new GraphQLClient(TRAY_GRAPHQL_ENDPOINT, {
  headers: { Authorization: `Bearer ${USER_TOKEN}` },
});

// List all workflows in the workspace
const LIST_WORKFLOWS = gql`
  query {
    viewer {
      workflows {
        edges {
          node {
            id
            name
            enabled
            triggerUrl
          }
        }
      }
    }
  }
`;

// Enable/disable a workflow
const UPDATE_WORKFLOW = gql`
  mutation ($workflowId: ID!, $enabled: Boolean!) {
    updateWorkflowStatus(
      input: { workflowId: $workflowId, enabled: $enabled }
    ) {
      workflow {
        id
        enabled
      }
    }
  }
`;

async function listWorkflows() {
  const data = await client.request(LIST_WORKFLOWS);
  return data.viewer.workflows.edges.map((e) => e.node);
}

async function toggleWorkflow(workflowId, enabled) {
  return client.request(UPDATE_WORKFLOW, { workflowId, enabled });
}

// Usage
const workflows = await listWorkflows();
console.log(`Found ${workflows.length} workflows`);
```

## Data Mapping

### ERP Integration Field Mapping Patterns

When connecting ERP systems through Tray.ai, field mapping happens in the Data Mapper step, which supports JSONPath expressions, JavaScript transforms, and lookup tables.

| Source Field | Target Field | Type | Transform | Gotcha |
|---|---|---|---|---|
| Salesforce: Account.Name | NetSuite: customer.companyName | String | Direct mapping | NetSuite max 83 chars vs Salesforce 255 |
| Salesforce: Opportunity.Amount | NetSuite: salesOrder.total | Currency | Currency conversion if multi-currency | Must set exchange rate subsidiary |
| SAP: VBAK-NETWR (net value) | Salesforce: Opportunity.Amount | Decimal | Divide by 100 (SAP stores in smallest unit) | SAP CURR fields store in minor currency units |
| NetSuite: transaction.tranDate | Salesforce: Opportunity.CloseDate | DateTime | Format conversion (ISO 8601 to YYYY-MM-DD) | NetSuite timezone = subsidiary timezone |
| Any: Multi-value field | Tray: JSON array | Array | Split/join with List Helpers | Step payload limit 6 MB for large arrays |

### Data Type Gotchas

- Tray.ai Data Mapper outputs JSON strings by default — numeric ERP fields (amounts, quantities) must be explicitly cast using JavaScript steps or the "Type Converter" helper to avoid string-to-number mismatches downstream. [src1]
- Date/time fields lose timezone information when passing through Tray workflow steps unless explicitly preserved — use ISO 8601 with timezone offset (e.g., `2026-03-03T10:00:00+01:00`) in all transformations. [src1]
- NetSuite multi-select custom fields serialize as pipe-delimited (`val1|val2`) in SuiteTalk but Tray's NetSuite connector returns them as arrays — branching on string vs array type is needed when the connector version changes. [src5]
- Null vs empty string handling differs between Tray connectors — Salesforce connector returns `null`, NetSuite connector returns `""` for empty fields. Use Boolean step to normalize before comparison logic. [src5]

## Error Handling & Failure Points

### Common Error Codes

| Code | Meaning | Cause | Resolution |
|---|---|---|---|
| HTTP 429 | Rate limit exceeded | >30 req/s to Tray API or high-volume webhooks | Exponential backoff: wait 2^n seconds; use "Auto respond 200" for webhooks |
| STEP_TIMEOUT | Connector operation timed out | Standard connector exceeded 45s timeout | Switch to HTTP Client (15-min timeout) or break into smaller operations |
| PAYLOAD_TOO_LARGE | Step data exceeds 6 MB | Large dataset between workflow steps | Use CSV connector, Data Storage, or chunk into smaller payloads |
| AUTH_REFRESH_FAILED | OAuth token refresh failed after 3 retries | Expired refresh token or revoked access | Re-authenticate the connector in Tray UI; check token lifetime on provider side |
| CONNECTOR_ERROR | Target system returned error | Invalid data, missing required fields, permission denied | Check step input mapping; verify connector auth permissions |
| WORKFLOW_DISABLED | Workflow triggered but not enabled | Workflow was disabled manually or via API | Enable workflow in Tray UI or via GraphQL mutation |

[src1, src2]

### Failure Points in Production

- **Task credit exhaustion mid-workflow**: Workflows stop executing when account task credits are depleted. There is no graceful mid-run handling — steps simply fail. Fix: `Set up credit usage alerts at 80% threshold; implement credit monitoring via Tray's usage API; design workflows to minimize step count per record`. [src7]
- **6 MB step payload overflow on growing datasets**: Works in dev with 100 records, fails in production with 10,000. The error message is not always obvious. Fix: `Use CSV connector for datasets >1,000 records; implement Data Storage for intermediate results; chunk payloads with List Helpers before processing`. [src1]
- **OAuth token expiration during long-running bulk workflows**: A 60-minute CSV processing job can outlast the access token. Tray auto-refreshes on 401 but only retries 3 times. Fix: `Pre-refresh tokens at workflow start; use service account auth (Client Credentials) instead of user-context auth for batch workflows`. [src1]
- **Webhook replay on Tray platform restarts**: During rare platform maintenance windows, webhooks can be replayed. Without idempotency in target ERP, this creates duplicate records. Fix: `Implement idempotency keys in workflow logic; use Data Storage to track processed event IDs; add deduplication step before ERP write operations`. [src1]
- **Connector version updates break field mappings**: When Tray updates a connector version, field names or response structures can change. Fix: `Pin connector versions in production workflows; test in a separate workspace before promoting; implement field-existence checks with Boolean steps`. [src5]

## Anti-Patterns

### Wrong: Processing large datasets record-by-record in a single loop

```javascript
// BAD — each record = N steps = N task credits; 10K records * 5 steps = 50K credits
// Also: 6 MB step limit will be hit as loop accumulates results
// Loop Connector:
//   For each record in $.steps.trigger.body.records (10,000 items):
//     Step 1: Look up customer in NetSuite
//     Step 2: Transform data
//     Step 3: Create/Update record in NetSuite
//     Step 4: Log result
// Total: 40,000 task credits, likely hits payload limit
```

### Correct: Chunk data and use callable workflows for parallel batch processing

```javascript
// GOOD — chunk into 200-record batches, process in parallel callable workflows
// Main Workflow:
//   Step 1: List Helpers - Chunk (size: 200) --> 50 chunks
//   Step 2: Loop over chunks
//     Step 2a: Call "Batch Processor" workflow with chunk as input
//
// "Batch Processor" Callable Workflow:
//   Step 1: NetSuite - Bulk Create/Update (batch operation, single API call)
//   Step 2: Return results
//
// Total: ~150 task credits (main) + 50 * 2 steps (child) = 250 credits
// 160x more efficient than record-by-record
```

### Wrong: Using synchronous webhook mode for high-volume event streams

```javascript
// BAD — "Await workflow and respond" mode rate-limits at high volume
// and has 5-minute timeout; hundreds of events/second triggers 429s
// Webhook trigger: "Await workflow and respond"
// + Complex 20-step workflow processing each event synchronously
// Result: 429 errors, dropped events, unhappy source systems
```

### Correct: Async webhook with queue pattern for reliable high-volume processing

```javascript
// GOOD — "Auto respond with status 200" (no rate limit) + async processing
// Webhook trigger: "Auto respond with status 200"
//   Step 1: Data Storage - Write event to queue (key: event_id, value: payload)
//   Step 2: Done (source system gets immediate 200)
//
// Separate scheduled workflow (runs every 1 min):
//   Step 1: Data Storage - Read all queued events
//   Step 2: Process batch
//   Step 3: Data Storage - Delete processed events
//   Step 4: Error handler for failed events (move to DLQ)
```

### Wrong: Hardcoding field mappings without null/type checks

```javascript
// BAD — assumes all fields exist and have expected types
// Data Mapper:
//   target.amount = $.steps.salesforce.Amount
//   target.name = $.steps.salesforce.Account.Name
//   target.date = $.steps.salesforce.CloseDate
// Fails when: Amount is null, Account is a deleted reference, CloseDate format changes
```

### Correct: Defensive mapping with Boolean checks and type conversion

```javascript
// GOOD — null checks + type conversion + fallback values
// Step 1: Boolean - Check if $.steps.salesforce.Amount is not null
//   TRUE:  Type Converter - Convert Amount to Number
//   FALSE: Set default value (0.00)
// Step 2: Boolean - Check if $.steps.salesforce.Account is not null
//   TRUE:  Map Account.Name
//   FALSE: Set to "Unknown Customer" + flag for review
// Step 3: Script step (JavaScript):
//   const closeDate = steps.salesforce.CloseDate;
//   return closeDate ? new Date(closeDate).toISOString() : new Date().toISOString();
```

## Common Pitfalls

- **Task credit math blindsides budget**: Developers build workflows in dev with 10 test records and deploy to production processing 100K records. Each step in a loop multiplies credits. A seemingly simple 5-step workflow processing 50K records/day = 250K credits/day = entire Pro tier allocation in 1 day. Fix: `Calculate task credits BEFORE deploying: (steps_per_record * daily_volume * runs_per_day). Use bulk connector operations to reduce step count per record`. [src7]
- **Connector timeouts on slow ERP APIs**: SAP and Oracle ERP Cloud APIs often respond in 10-30 seconds for complex queries. The default 45-second connector timeout leaves no margin. Fix: `Use the HTTP Client connector (15-min timeout) instead of native connectors for slow endpoints; implement async polling patterns for long-running ERP operations`. [src1]
- **Data Storage scope confusion**: "Current Run" scope data persists for 30 days, not just the current execution. Developers assume it's cleared between runs and get stale data. Fix: `Explicitly delete Data Storage keys at workflow end; use unique run-scoped keys (e.g., prefix with execution_id)`. [src1]
- **Workspace isolation breaks cross-team workflows**: Auth credentials are workspace-scoped. A workflow moved to a different workspace loses all connector authentications. Fix: `Plan workspace architecture before building; use Tray's solution packaging for cross-workspace deployment; re-authenticate all connectors after workspace moves`. [src6]
- **CSV connector silently truncates at 8 KB per row**: Wide ERP records (SAP material masters with 200+ fields) can exceed the per-row limit. No error is thrown — data is silently truncated. Fix: `Split wide records into multiple CSVs by field group; verify output row lengths in testing; use JSON-based Data Storage instead of CSV for wide records`. [src1]
- **Merlin AI generates workflows that don't handle errors**: AI-generated workflows from natural language prompts typically lack error handling, retry logic, and edge case handling. Fix: `Always review and augment Merlin-generated workflows; add error handling branches manually; test with invalid data before deploying`. [src4]

### Deprecation Policy

Tray.ai follows a continuous release model with no numbered API versions. Platform changes are deployed continuously, with breaking changes communicated via in-app notifications and release notes. Connector updates are generally backward-compatible, but major connector version bumps (e.g., NetSuite connector v2 to v3) may change response schemas. Tray maintains backward compatibility for at least 12 months on major connector versions. [src3]

## Important Caveats

- Task credit pricing is opaque — Tray does not publicly disclose per-credit costs or overage rates. Negotiate credit volume and overage pricing upfront during contract discussions. [src7]
- The 6 MB inter-step payload limit is the most commonly hit constraint in ERP integrations and is not obvious during development with small test datasets. Always test with production-scale data volumes. [src1]
- No native SAP connector means SAP integrations require significantly more development effort (HTTP Client configuration, custom auth, response parsing) compared to Workato or MuleSoft alternatives. [src5]
- Tray.ai is cloud-only with no on-premise deployment option. Organizations in regulated industries (banking, healthcare, government) must verify that data transit through Tray's AWS infrastructure meets their compliance requirements. [src6]
- Connector updates are pushed automatically — there is no way to pin a specific connector version. Test critical workflows after any connector update notification. [src5]
- Merlin AI workflow generation is a starting point, not production-ready — always add error handling, logging, and edge case management to AI-generated workflows. [src4]
