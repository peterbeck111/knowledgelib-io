# Oracle Redwood UI Migration: Impact on Custom Pages, Analytics, and VBCS Extensions

## TL;DR

- **Bottom line**: Oracle Redwood replaces the Classic UI (ADF/Responsive) across all Fusion Cloud modules by 26D/27A — existing sandbox personalizations do NOT auto-migrate and must be manually recreated in Visual Builder Studio (VBS), while VBCS extensions require theme migration from Alta to Redwood.
- **Key limit**: Classic UI is formally deprecated in update 26D (November 2026) and fully decommissioned in 27A (February 2027) — organizations on Classic UI face a hard deadline.
- **Watch out for**: Sandbox personalizations (page layout changes, field visibility, conditional rendering) made through the traditional sandbox interface are completely lost when a page switches to Redwood — the Personalization Helper Tool helps identify them but does not convert them automatically.
- **Best for**: Organizations running Oracle Fusion Cloud that need to assess, plan, and execute the Redwood UI transition while preserving existing customizations and extensions.
- **Authentication**: No authentication changes — Redwood is a UI layer change that uses the same underlying REST APIs, security model, and role-based access control.

This card covers the impact of Oracle's Redwood Experience transition on existing Oracle Fusion Cloud customizations, extensions, analytics dashboards, and integration points that touch the UI layer. It addresses the migration timeline by module, the fate of existing sandbox personalizations, VBCS extension migration from Alta to Redwood theme, analytics and reporting implications, and strategies for maintaining continuity during the transition. ## API Surfaces & Capabilities

Redwood is primarily a UI/UX transformation — it does not change the underlying REST APIs, SOAP services, or integration endpoints. However, it significantly affects the UI extension layer and any integrations that interact with the page surface. [src2]

| Component | Classic UI (Before) | Redwood UI (After) | Integration Impact | Action Required |
|---|---|---|---|---|
| Page Personalizations | Sandbox (ADF Page Composer) | VBS (Visual Builder Studio) | High | Recreate manually in VBS |
| Custom Pages | VBCS with Alta theme | VBCS with Redwood theme | Medium | Theme migration required |
| Embedded Analytics | OTBI/BIP in ADF pages | OTBI/BIP in Redwood pages | Low-Medium | Verify rendering in Redwood |
| Custom Branding/CSS | ADF skin customization | Redwood theme + stable theme override | Medium | Redesign CSS approach |
| Deep Links / URLs | ADF page URLs | Redwood page URLs | Medium | Update bookmarks and integration links |
| REST API Endpoints | Unchanged | Unchanged | None | No action needed |
| SOAP Services | Unchanged | Unchanged | None | No action needed |
| FBDI/ADFdi | Unchanged | Unchanged | None | No action needed |

## Authentication

Redwood does not change the authentication model. All existing authentication flows (SAML 2.0, IDCS, OAuth 2.0) remain unchanged. However, there are implications for user session behavior. [src4]

| Aspect | Classic UI | Redwood UI | Impact |
|---|---|---|---|
| Authentication flow | SAML 2.0 / IDCS | Same | No change |
| Session management | ADF session | JET session | Session timeout behavior may differ |
| Role-based access | Data roles + job roles | Same | No change to security model |
| Deep link authentication | ADF parameter-based | Redwood URL structure | URL format changes may affect SSO redirects |

### Authentication Gotchas

- Deep links that pass authentication context via ADF URL parameters may need updating for Redwood URL structure — test all SSO redirect flows after enabling Redwood on a page [src4]
- Browser bookmarks to Classic UI pages will redirect to Redwood equivalents — but custom integration links that construct URLs programmatically may break [src3]
- VBS sandbox access requires additional roles (Visual Builder Administrator or Application Developer) — users who could personalize via Sandbox may need new roles for VBS [src1]

## Constraints

- Sandbox personalizations made in Classic UI do NOT auto-migrate to Redwood — every personalization must be recreated in VBS [src1, src5]
- Not all Classic UI pages have Redwood equivalents yet — check the Redwood Availability Matrix per module and release [src6]
- VBCS extensions using Alta theme will not visually match Redwood pages — theme migration is required for visual consistency [src2]
- Custom CSS overrides of the Redwood theme may break on future Oracle updates — use the stable theme base to protect against core theme changes [src1]
- Oracle pushes Redwood by module on a rolling quarterly schedule — organizations cannot opt out once a module's Redwood pages are delivered [src4]
- The Redwood Personalization Helper Tool identifies existing personalizations but does not automatically convert them to VBS format — it produces a report that must be manually acted upon [src5]

## Step-by-Step Integration Guide

### 1. Run Redwood Personalization Helper Tool

Use Oracle's Helper Tool to generate a comprehensive report of all existing sandbox personalizations that need migration. [src5]

```
# Access the Redwood Personalization Helper Tool:
# 1. Navigate to: Setup and Maintenance → Manage Sandboxes
# 2. Open the Personalization Helper Tool (available from 24B+)
# 3. Select modules to scan (HCM, ERP, SCM, CX)
# 4. Run scan — generates CSV/PDF report listing:
#    - Page name
#    - Personalization type (layout, visibility, conditional)
#    - Complexity rating (simple/moderate/complex)
#    - Compatible with Redwood auto-migration? (Yes/No)
# 5. Download report for migration planning
```

**Verify**: Report covers all modules with active personalizations — compare count against known customization inventory.

### 2. Migrate Sandbox Personalizations to VBS

For each personalization identified by the Helper Tool, recreate it in Visual Builder Studio using Redwood components. [src1, src5]

```
# VBS Personalization Migration Process:
# 1. Open Visual Builder Studio → Application Extensions
# 2. Create a new sandbox for migration work
# 3. Navigate to the Redwood page equivalent of the Classic page
# 4. Use Page Designer to apply equivalent personalizations:
#    - Field visibility → Use "Visible" property with condition
#    - Layout changes → Drag-and-drop in Page Designer
#    - Business rules → Create Rule in VBS (replaces Groovy)
# 5. Test in VBS Preview mode on desktop and mobile
# 6. Publish sandbox → Test in non-production environment
# 7. Promote to production via Change Management
```

**Verify**: Open the Redwood page — verify all personalizations are visible and functional. Compare screenshot against Classic UI version.

### 3. Migrate VBCS Extensions from Alta to Redwood Theme

Update existing VBCS custom applications to use the Redwood design system instead of Alta. [src2]

```javascript
// Step 1: Update app-flow.json to use Redwood theme
// Before (Alta):
{
  "theme": "alta",
  "requirejs": { "paths": { "ojs": "libs/oracle/jet" } }
}

// After (Redwood):
{
  "theme": "redwood",
  "requirejs": { "paths": { "ojs": "libs/oracle/jet" } }
}

// Step 2: Update component references
// Replace Alta-specific components with Redwood equivalents:
// - oj-button (Alta) → oj-button (Redwood — same component, different styling)
// - Custom CSS classes → Use Redwood design tokens
// - Alta color variables → Redwood color tokens

// Step 3: Create stable theme override for custom branding
// In web app settings → Redwood theme → Create CSS override file
// Use stable theme base to protect against Oracle's future theme changes
```

**Verify**: Preview VBCS extension in Redwood mode — verify visual consistency with Redwood Fusion pages on desktop and mobile.

### 4. Update Deep Links and Integration URLs

Identify and update all external references to Fusion Cloud page URLs that may change with Redwood. [src3]

```bash
# Audit deep links in external systems:
# 1. Search email templates for ADF URLs:
#    Pattern: https://<pod>.fa.us2.oraclecloud.com/fscmUI/faces/...
# 2. Search integration flows (OIC) for UI-level URLs
# 3. Search documentation/wikis for bookmarked Fusion pages
# 4. Search SSO/SAML configurations for redirect URLs
#
# Redwood URL format:
#    https://<pod>.fa.us2.oraclecloud.com/fscmUI/redwood/...
# Classic redirect behavior:
#    Classic URLs redirect to Redwood equivalents
#    BUT: URL parameters may not be preserved
#    TEST each deep link after Redwood enablement
```

**Verify**: Each deep link resolves correctly and preserves user context (record ID, parameters) in the Redwood page.

## Code Examples

### JavaScript: VBCS Extension Theme Migration

```javascript
// Input:  Existing VBCS extension with Alta theme
// Output: Updated extension with Redwood theme

// Example: Migrating a custom dashboard component
// Before (Alta theme):
define(['ojs/ojcore', 'knockout', 'ojs/ojbutton'],
  function(oj, ko) {
    function DashboardViewModel() {
      this.buttonClass = ko.observable('oj-button-primary alta-button');
      // Alta-specific styling
      this.panelClass = 'alta-panel oj-panel-shadow';
    }
    return DashboardViewModel;
  });

// After (Redwood theme):
define(['ojs/ojcore', 'knockout', 'ojs/ojbutton'],
  function(oj, ko) {
    function DashboardViewModel() {
      // Redwood uses design tokens instead of Alta classes
      this.buttonClass = ko.observable('oj-button-primary');
      // Redwood panel styling — use design tokens
      this.panelClass = 'oj-panel oj-panel-shadow-sm';
    }
    return DashboardViewModel;
  });
```

## Error Handling & Failure Points

### Common Error Codes

| Code | Context | Meaning | Cause | Resolution |
|---|---|---|---|---|
| VBS-SANDBOX-001 | VBS | Sandbox creation failed | Max concurrent sandboxes reached | Close unused sandboxes; wait for expiry |
| VBS-DEPLOY-403 | VBS | Deployment forbidden | Missing Visual Builder Administrator role | Grant role to user in Security Console |
| REDWOOD-PAGE-404 | UI | Redwood page not available | Page not yet converted to Redwood in current release | Stay on Classic for this page until next release |
| CSS-OVERRIDE-BROKEN | UI | Custom styling lost after update | Used unstable Redwood theme class names | Switch to stable theme base; use design tokens |
| DEEP-LINK-REDIRECT | Integration | URL redirect loses parameters | Classic URL format not fully compatible | Update URL to Redwood format |

### Failure Points in Production

- **Personalizations lost after Redwood enablement**: Users report missing fields and layout changes when a page switches to Redwood. Fix: `Run Personalization Helper Tool before enabling Redwood on any page; recreate all personalizations in VBS first`. [src5]
- **VBCS extension visual mismatch**: Custom VBCS pages still show Alta styling while surrounding Fusion pages show Redwood. Fix: `Migrate VBCS extensions to Redwood theme before enabling Redwood on the containing module`. [src2]
- **Custom CSS breaks on quarterly update**: Oracle updates Redwood theme internals, breaking custom CSS that relied on internal class names. Fix: `Use stable theme base and Redwood design tokens only; never reference internal oj-* class names directly`. [src1]
- **Email template deep links stop working**: Hardcoded Classic UI URLs in notification templates redirect but lose record context. Fix: `Update all email/notification templates to use Redwood URL format; test each template after Redwood enablement`. [src3]

## Anti-Patterns

### Wrong: Assuming personalizations auto-migrate to Redwood

```
# BAD — Enabling Redwood on a module and expecting personalizations to appear
# 1. Enable Redwood on Financials pages
# 2. Users log in → custom fields and layouts are GONE
# 3. Users report "my customizations disappeared"
# This is by design — Classic personalizations do NOT migrate
```

### Correct: Migrate personalizations BEFORE enabling Redwood

```
# GOOD — Migration-first approach:
# 1. Run Personalization Helper Tool → Get report of all customizations
# 2. Recreate each personalization in VBS sandbox
# 3. Test in VBS Preview mode
# 4. Publish VBS sandbox
# 5. THEN enable Redwood on the module pages
# 6. Users see customizations + Redwood UI simultaneously
```

### Wrong: Overriding Redwood CSS with hardcoded class names

```css
/* BAD — These internal class names change between Oracle updates */
.oj-navigationlist-item-element {
  background-color: #1a73e8 !important; /* Will break */
}
.oj-flex-bar-end .oj-button-text {
  color: white !important; /* Will break */
}
```

### Correct: Use Redwood design tokens and stable theme

```css
/* GOOD — Design tokens are stable across updates */
:root {
  --oj-core-brand-color: #1a73e8;
}
.oj-web-applayout-header {
  background-color: var(--oj-header-bg-color);
}
/* Use stable theme base — Oracle commits to backward compatibility */
```

### Wrong: Delaying Redwood migration until the deadline

```
# BAD — Waiting until 26D (November 2026) to start migration
```

### Correct: Start phased migration aligned with module rollout

## Common Pitfalls

- **Not running the Personalization Helper Tool first**: Organizations enable Redwood without knowing what personalizations exist, then face user complaints about missing customizations. Fix: `Run the Helper Tool for every module before enabling Redwood; export the report as your migration checklist`. [src5]
- **Mixing Alta and Redwood themes in VBCS**: Some VBCS pages use Alta while Redwood is enabled on surrounding Fusion pages, creating a jarring visual experience. Fix: `Migrate all VBCS extensions to Redwood theme before enabling Redwood on the parent module`. [src2]
- **Using unstable CSS overrides**: Custom CSS that references Oracle's internal class names breaks on quarterly updates. Fix: `Use Redwood design tokens exclusively; switch to stable theme base for any CSS overrides`. [src1]
- **Ignoring deep link updates**: Email templates, bookmarks, and integration links that construct Classic UI URLs stop working correctly. Fix: `Audit all deep links across email templates, notification configs, OIC flows, and documentation; update to Redwood URL format`. [src3]
- **Not budgeting for regression testing**: Each module's Redwood migration requires full regression testing of all personalizations, extensions, and analytics. Fix: `Budget 1-2 weeks per module for regression testing; automate tests where possible using Selenium or similar`. [src4]

### Deprecation Policy

Oracle has committed to the Redwood transition with a firm timeline. Classic UI (ADF/Responsive) is deprecated in 26D and decommissioned in 27A. Oracle will not extend Classic UI support beyond 27A. All sandbox personalizations must be migrated to VBS before 27A. VBCS extensions on Alta theme will still function but will appear visually inconsistent with Redwood pages. [src4, src6]

## Important Caveats

- Redwood is a mandatory transition — there is no "opt-out" option after a module's Redwood pages are delivered
- Classic UI is decommissioned in 27A (February 2027) — this is a hard deadline, not a soft suggestion
- The Personalization Helper Tool is useful but incomplete — it does not detect custom page sections, embedded analytics, or VBCS extensions
- Oracle's Redwood timeline has been revised multiple times — track Oracle Cloud Readiness for the latest dates
- VBCS extensions on Alta theme will technically still work after Redwood enablement but will create a poor user experience due to visual inconsistency
- Custom CSS overrides must use Redwood design tokens and stable theme base — any direct class name references will break on future updates
