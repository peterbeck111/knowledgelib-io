# Executive Command Center Design

## TL;DR

- **Bottom line**: An executive command center is a single-screen dashboard showing 8-12 KPIs with RAG status indicators, automated alert triggers, and action rules that turn metrics into decisions.
- **Key tool/command**: RAG (Red/Amber/Green) threshold system with measurable boundaries — e.g., "Green if MRR growth > 10%, Amber if 5-10%, Red if < 5%"
- **Watch out for**: Dashboard overload — more than 12 KPIs reduces actionability to zero. Ruthlessly cut to the metrics that drive decisions.
- **Works with**: Any BI tool (Metabase, Grafana, Preset, Tableau, Looker) or custom-built with React + charting library.

## Constraints
<!-- Agents: read this section before recommending any action from this unit. -->

- Maximum 8-12 KPIs per dashboard screen. More overwhelms decision-makers. [src1]
- Every KPI must have a single accountable owner with a defined escalation path. Unowned KPIs are vanity metrics.
- RAG thresholds must use measurable data points, not subjective judgment. "Green if within 5% of target" not "green if it feels good." [src3]
- Limit automated alerts to critical events (red status only). Alerting on amber creates alert fatigue that causes executives to ignore real problems. [src5]
- Dashboard must load in under 3 seconds. Executives will not wait for slow analytics queries.
- Include accessibility considerations: use symbols alongside colors for color-blind users (2.7 million in UK alone). [src5]

## Decision Tree

```
START: What type of command center?
├── Audience is CEO/founder only?
│   ├── YES → Build a single-screen overview with 5-8 KPIs covering all functions
│   └── NO ↓
├── Audience is executive team?
│   ├── YES → Build overview + department drill-downs (one tab per function)
│   └── NO ↓
├── Audience is board of directors?
│   ├── YES → Build quarterly snapshot with trend comparisons (simpler, higher-level)
│   └── NO ↓
└── Company-wide all-hands?
    └── Build simplified version with 3-5 company-level KPIs (no sensitive data)

KPI Selection Framework:
├── Does this metric directly measure progress toward a company goal?
│   ├── NO → Remove it
│   └── YES ↓
├── Can an owner take action when this metric changes?
│   ├── NO → It is an FYI metric, move to drill-down page
│   └── YES ↓
├── Does this metric change frequently enough to warrant dashboard space?
│   ├── NO → Move to monthly/quarterly report
│   └── YES → Include on command center
```

## Step-by-Step Guide

### 1. Select KPIs Using the 3-Filter Method

Apply three filters to every candidate KPI to decide if it belongs on the command center: [src1] [src2]

**Filter 1: Strategic alignment** — Does this KPI directly measure progress toward one of the company's top 3-5 goals? If not, remove it.

**Filter 2: Actionability** — When this KPI changes, can a specific person take a specific action? If the answer is "we'd look into it" rather than "VP Sales would do X," it is not actionable enough.

**Filter 3: Volatility** — Does this KPI change frequently enough to warrant dashboard space? Metrics that move quarterly belong in quarterly reports, not real-time dashboards.

**Recommended startup command center KPIs (choose 8-12):**

| Category | KPI | Leading/Lagging | Update Frequency |
|----------|-----|-----------------|------------------|
| Revenue | MRR / ARR | Lagging | Daily |
| Revenue | MRR Growth Rate | Leading | Weekly |
| Revenue | Net Revenue Retention | Lagging | Monthly |
| Acquisition | CAC (Customer Acquisition Cost) | Lagging | Monthly |
| Acquisition | Pipeline Value | Leading | Daily |
| Retention | Churn Rate | Lagging | Monthly |
| Retention | NPS / CSAT | Leading | Weekly |
| Product | DAU/MAU Ratio | Leading | Daily |
| Product | Feature Adoption Rate | Leading | Weekly |
| Efficiency | Burn Rate / Runway | Lagging | Monthly |
| Efficiency | CAC Payback Period | Lagging | Monthly |
| Team | eNPS (Employee Net Promoter Score) | Leading | Monthly |

**Verify**: Selected 8-12 KPIs, each passes all 3 filters, mix of leading and lagging indicators.

### 2. Define RAG Thresholds for Each KPI

For every KPI, define measurable thresholds for Green, Amber, and Red status: [src3] [src5]

**Threshold design rules:**
- Green = on track or exceeding target (no action needed)
- Amber = caution, trending toward risk (owner should monitor closely)
- Red = off track, requires immediate action (trigger alert + action rule)

**Example threshold definitions:**

| KPI | Green | Amber | Red |
|-----|-------|-------|-----|
| MRR Growth | > 10% MoM | 5-10% MoM | < 5% MoM |
| Churn Rate | < 2% monthly | 2-5% monthly | > 5% monthly |
| Burn Rate | > 18 months runway | 12-18 months runway | < 12 months runway |
| NPS | > 50 | 30-50 | < 30 |
| CAC Payback | < 12 months | 12-18 months | > 18 months |
| Pipeline Coverage | > 3x quota | 2-3x quota | < 2x quota |

**Rules for good thresholds:**
- Use percentages or ratios, not absolute numbers (they scale with company growth)
- Set amber as a buffer zone — wide enough to provide early warning, narrow enough to be meaningful
- Review and adjust thresholds quarterly as the company grows
- Document the rationale for each threshold so new executives understand the logic

**Verify**: Every KPI has Green/Amber/Red thresholds with measurable data points.

### 3. Design the Alert and Notification System

Create an alert framework that notifies the right people without creating alert fatigue: [src4] [src6]

**Alert tiers:**

| Tier | Trigger | Notification Channel | Response Time | Example |
|------|---------|---------------------|---------------|---------|
| Critical (Red) | KPI crosses red threshold | Slack DM + Email to owner + dashboard badge | < 4 hours | Churn > 5%, Runway < 12 months |
| Warning (Amber) | KPI crosses amber threshold | Dashboard badge only (no push notification) | Next business day | MRR growth 5-10%, NPS 30-50 |
| Info | Notable change (> 2 standard deviations) | Weekly digest email | Weekly review | Unusual traffic spike, new competitor pricing |

**Alert design rules:**
- Never alert on amber — this creates noise that trains people to ignore alerts
- Attach a 3-step remediation checklist to every critical alert
- Include context: current value, threshold, trend direction, last 4 data points
- Deduplicate: if a KPI has been red for 3 consecutive days, do not send 3 alerts
- Auto-escalate: if red status persists > 48 hours without acknowledgment, escalate to CEO

**Verify**: Alert tiers defined, notification channels configured, remediation checklists attached to each critical alert.

### 4. Build Action Rules for Each Red KPI

Every red KPI must have a pre-defined action rule — what happens when the metric goes critical: [src2]

**Action rule template:**

```
IF [KPI] crosses RED threshold:
  THEN [Owner] must:
    1. [Immediate action — diagnose root cause within N hours]
    2. [Corrective action — implement fix within N days]
    3. [Report action — update team in next standup/channel]
  ESCALATE TO [Executive] if not resolved within [timeframe]
```

**Example action rules:**

| KPI in Red | Owner | Immediate Action | Corrective Action | Escalation |
|------------|-------|-----------------|-------------------|------------|
| Churn > 5% | VP CS | Review top 10 at-risk accounts within 4 hours | Implement retention campaign within 48 hours | CEO if > 5 days |
| MRR growth < 5% | VP Sales | Analyze pipeline conversion by stage | Adjust sales targets or lead gen spend | CEO if 2 consecutive months |
| Runway < 12 months | CFO | Model scenarios: cut burn or raise | Present options to CEO within 24 hours | Board if < 9 months |
| NPS < 30 | VP Product | Categorize detractor feedback themes | Ship top 3 fixes within 2 sprints | CEO if persists 30 days |

**Verify**: Every red-threshold KPI has an action rule with owner, timeline, and escalation path.

### 5. Design the Dashboard Layout

Arrange the command center following visual hierarchy principles: [src1] [src6]

**Layout structure (single screen):**

```
┌─────────────────────────────────────────────────┐
│  HEADER: Company Name | Last Updated | Alerts(N)│
├─────────────────────────────────────────────────┤
│  ROW 1: Top-level financial KPIs (3-4 cards)    │
│  [MRR + trend] [Growth %] [Runway] [Burn Rate]  │
├─────────────────────────────────────────────────┤
│  ROW 2: Customer/product KPIs (3-4 cards)       │
│  [Churn] [NPS] [DAU/MAU] [Pipeline]             │
├─────────────────────────────────────────────────┤
│  ROW 3: Operational KPIs (2-3 cards)            │
│  [CAC] [CAC Payback] [eNPS]                     │
├─────────────────────────────────────────────────┤
│  ROW 4: Action Items (red KPIs with next steps) │
│  [Alert 1: owner + action] [Alert 2: ...]       │
└─────────────────────────────────────────────────┘
```

**Visual design rules:**
- Most important KPIs in top-left (natural reading order)
- Each KPI card: large number, small sparkline, RAG dot, owner initials
- Use consistent card sizes — visual irregularity creates cognitive load
- White space between sections — crowded dashboards are useless
- No pie charts on executive dashboards — they are hard to compare accurately
- Use symbols alongside colors for accessibility (checkmark, warning triangle, X)

**Verify**: Layout fits on single screen, follows visual hierarchy, all KPIs visible without scrolling.

## Code Examples

### React: KPI Card Component

```jsx
// Input:  KPI name, value, trend, threshold config
// Output: Styled card with RAG indicator and sparkline

function KPICard({ name, value, trend, thresholds, owner }) {
  const status = value >= thresholds.green ? 'green'
    : value >= thresholds.amber ? 'amber' : 'red';
  const icon = status === 'green' ? '✓' : status === 'amber' ? '⚠' : '✗';
  return (
    <div className={`kpi-card kpi-${status}`}>
      <span className="kpi-status" aria-label={status}>{icon}</span>
      <h3>{name}</h3>
      <p className="kpi-value">{value}</p>
      <Sparkline data={trend} />
      <span className="kpi-owner">{owner}</span>
    </div>
  );
}
```

### Python: Alert Threshold Evaluator

```python
# Input:  KPI data dict with current values and threshold config
# Output: List of alerts for KPIs that crossed red threshold

def evaluate_alerts(kpis, thresholds):
    alerts = []
    for kpi_name, current in kpis.items():
        config = thresholds[kpi_name]
        if config["direction"] == "higher_is_better":
            if current < config["red"]:
                alerts.append({"kpi": kpi_name, "value": current,
                    "threshold": config["red"], "severity": "critical"})
        else:  # lower_is_better (e.g., churn)
            if current > config["red"]:
                alerts.append({"kpi": kpi_name, "value": current,
                    "threshold": config["red"], "severity": "critical"})
    return alerts
```

## Anti-Patterns

### Wrong: Dashboard with 30+ KPIs on one screen
Showing every available metric on a single dashboard because stakeholders each want "their" metric visible. Result: executives glance at it for 2 seconds, absorb nothing, and stop using the dashboard entirely. [src1]

### Correct: 8-12 KPIs with drill-down
Ruthlessly limit the command center to 8-12 KPIs that pass the 3-filter test. Everything else goes on department-specific drill-down pages accessible with one click from the main dashboard.

### Wrong: Subjective RAG assignment
Letting project managers or team leads manually assign green/amber/red based on their feeling. Result: everything is green until it is suddenly, catastrophically red. The amber zone is never used because no one wants to be the first to flag a problem.

### Correct: Data-driven RAG thresholds
Define measurable thresholds before launching the dashboard. Green/amber/red are calculated automatically from data. No human judgment in status assignment. Review thresholds quarterly. [src3]

### Wrong: Alerting on everything
Sending Slack notifications for every amber and red KPI change. Within a week, the team mutes the channel. Within a month, a real critical alert is missed because everyone learned to ignore notifications.

### Correct: Alert only on red, digest for amber
Reserve push notifications for red-threshold crossings only. Deliver amber status changes in a weekly digest email. This preserves the signal value of alerts. [src5]

## Common Pitfalls

- **Vanity metrics on the dashboard**: Showing total users instead of active users, total revenue instead of growth rate. Fix: every KPI must answer "so what?" — if it does not drive a decision, remove it. [src2]
- **No data freshness indicator**: Executives make decisions on stale data without knowing it. Fix: show "last updated" timestamp prominently on every card. Highlight in red if data is > 24 hours old. [src4]
- **Building before defining KPIs**: Jumping to Grafana/Metabase setup before deciding what to measure. Fix: complete Steps 1-4 (KPI selection, thresholds, alerts, action rules) on paper before touching any tool.
- **One-size-fits-all dashboard**: Same dashboard for CEO, board, and all-hands. Fix: create 2-3 views with different KPI subsets and detail levels for each audience. [src6]
- **No owner accountability**: KPIs exist on the dashboard but nobody is responsible when they go red. Fix: every KPI card displays the owner's name. Red KPI without acknowledged action within 4 hours triggers escalation. [src2]

## Important Caveats

- RAG thresholds should be reviewed and adjusted quarterly as the company grows — what is "red" at $100K MRR is different from what is "red" at $1M MRR
- Alert systems require ongoing tuning — start conservative (fewer alerts) and add gradually based on missed incidents
- Dashboard adoption requires executive sponsorship — if the CEO does not use it in weekly meetings, nobody else will either
- Consider time zones for alert routing if the team is distributed across geographies
