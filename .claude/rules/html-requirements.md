# HTML Requirements

## Required Meta Tags
- `<meta name="viewport" content="width=device-width, initial-scale=1">`
- Open Graph: `og:type` ("article"), `og:title`, `og:description` (same as meta description), `og:url`, `og:site_name` ("knowledgelib.io"), `og:image` ("https://knowledgelib.io/images/og-default.png"), `og:locale` ("en_US"), `article:published_time`, `article:modified_time`
- Twitter Card: `twitter:card` ("summary_large_image"), `twitter:title`, `twitter:description`, `twitter:image` (same og-default.png)
- Core `ai:*` meta tags in `<head>` (all templates): type, confidence, last_verified, domain, canonical_question, aliases, raw, token_estimate, source_count, freshness, version
- Additional `ai:*` meta tags (added by improvement plan):
  - `ai:entity_type` — `product_comparison`, `software_reference`, `fact`, `concept`, `rule`
  - `ai:jurisdiction` — `global`, `US`, `EU`, `UK`, etc.
  - `ai:temporal_status` — `stable`, `evolving`, `volatile`
  - `ai:change_sensitivity` — `low`, `medium`, `high`
  - `ai:constraints` — semicolon-separated hard limits from frontmatter constraints array (agents parse before body)
  - `ai:skip_this_unit_if` — semicolon-separated "condition → use_instead" pairs from frontmatter (negative constraints)
  - `ai:sections` — comma-separated list of available sections (software + concept + rule cards)
  - `ai:applies_to` — scope string for rule cards (e.g., "consumer_electronics > audio > wireless_earbuds; $0-$100")
  - `ai:inputs_needed` — comma-separated input keys when card has `inputs_needed` frontmatter (product + software cards)

## Universal Commerce Protocol meta tags (all unit pages)
Agent-readiness signals so LLM shopping agents can identify the page's commerce surface area:
- `<meta name="commerce:protocol" content="schema.org-offer; ucp">`
- `<meta name="commerce:agent-ready" content="true">`
- `<meta name="commerce:checkout-via" content="affiliate-redirect">` (or `"none"` for non-product cards)
- `<meta name="commerce:offers-count" content="N">` (count of `buy_links`, `0` for non-product cards)
- `<meta name="commerce:currency" content="USD">` (product cards only)

## Body Structure

### Canonical-Question H2 (every entity type)
Required immediately after `<h1>...</h1>` on every unit page:
```html
<h2 id="canonical-question-heading" class="canonical-question">{canonical_question — verbatim from frontmatter}</h2>
```
AI tree-walking algorithms (ChatGPT, Perplexity, Google AI Overviews) read H1 then H2 first. The canonical question phrased the way users ask it gives them an immediate question→answer pair before any prose. Copy verbatim from frontmatter — do not paraphrase.

### TL;DR section (product_comparison only)
Required between the canonical-question H2 and Summary:
```html
<section id="tldr" aria-labelledby="tldr-heading">
  <h2 id="tldr-heading">TL;DR</h2>
  <p><strong>Top pick: {Model} (~${price})</strong> — {one-clause why}.<br>
     <strong>Best value: ...</strong><br>
     <strong>Best budget: ...</strong></p>
</section>
```
30–50 words. Each bolded pick must be a buy_link product. RAG snippet extractors lift this verbatim — do not bury the answer.

### Head-to-Head Comparisons (product_comparison only)
Required between Best for Each Use Case and Decision Logic:
```html
<section id="head-to-head" aria-labelledby="head-to-head-heading">
  <h2 id="head-to-head-heading">Head-to-Head Comparisons</h2>
  <h3>{Model A} vs {Model B}</h3>
  <p>{verdict 2-3 sentences} [src]</p>
  <p><strong>Pick A if:</strong> ...<br><strong>Pick B if:</strong> ...</p>
  ...
</section>
```
3–5 pairwise matchups of the top picks. Use natural buyer comparisons, not every permutation. Captures "X vs Y" queries (~33% of all AI citations per Princeton GEO research).

### `ai:sections` meta tag (product_comparison)
Must include `tldr` and `head_to_head`. Full list:
```html
<meta name="ai:sections" content="tldr,summary,comparison_table,use_cases,head_to_head,decision_logic,market_trends,caveats">
```

## Sibling pricing.md (product_comparison only)

Required: alongside the .md and .html, a `pricing.md` file in the same directory listing every buy_link product with name, price (parsed from comparison table), ASIN, and `/go/{slug}` URL. Generated by `node knowledge_pipeline/generate_pricing_md.js`. AI shopping agents parse this without rendering the page.

## Schema.org JSON-LD
Required on every unit page:
- `Dataset` block (must include `"creator"` field — Google Search Console requires it)
- `BreadcrumbList` block (Home > Category > Subcategory > Title)
- `FAQPage` block — `mainEntity` is the canonical_question + first 5 aliases as `Question`/`Answer` pairs, all sharing the same `acceptedAnswer.text` derived from the primary section (Summary / TL;DR / Definition / Rule / Fact)

Do **not** add `aggregateRating` to `Dataset`, `TechArticle`, `DefinedTerm`, or any non-Product type. Google Search Console rejects it as "Invalid object type for field <parent_node>" because Review snippets only support Product, Book, Movie, Recipe, Event, Course, and similar types. Editorial confidence is exposed via `<meta name="ai:confidence">` and the visible badge in the unit-meta strip.

Type-specific:
- `TechArticle` block — software_reference cards (in addition to Dataset)
- `DefinedTerm` block — concept cards (in addition to Dataset)
- `Product` + `Offer` + `Review` `@graph` block — product_comparison cards only, one entry per `buy_link`. Each Product carries `name`, `url` (the `/go/{slug}` redirect), `sku` (ASIN), an `Offer` with `price` (extracted from comparison table), `priceCurrency: "USD"`, `availability: "InStock"`, `seller: Amazon`, plus a `Review` whose `reviewBody` is the first sentence of the matching use-case description and whose `author` is `knowledgelib.io`

## ARIA / Accessibility
Required on every unit page:
- `<main role="main">` wraps the page body content (between `<body>` open and `<footer>`)
- `<aside class="unit-meta" aria-label="Knowledge unit metadata">` instead of `<div>` for the metadata strip
- `<span class="confidence-badge" aria-label="Confidence score X out of 1">` on every confidence badge
- `<table aria-label="...">` with `<th scope="col">` headers on every table
- `<nav class="related-links" aria-label="Related knowledge units">` instead of `<div>` for the related-units block
- `<footer role="contentinfo" aria-label="Site footer">` on the page footer
- Buy links: `<a ... aria-label="Check price for {Product Name} on Amazon">` on every `/go/{slug}` link

## Other
- Affiliate links use `rel="sponsored nofollow"`
- Include `verified-date.js` script block with Supabase config
- Content is rendered HTML, never raw markdown
- **Default OG image**: `prototype/images/og-default.png` (1200x630, branded). Source SVG at `og-default.svg`

## Migration script
`knowledge_pipeline/migrate_agent_ready.js` — idempotent migration that injects all of the above into every existing card. Run after any bulk template change. Validate with `knowledge_pipeline/validate_agent_ready.js`.

## Static Pages

Static pages (`about.html`, `methodology.html`, `api.html`) live as flat files in `prototype/` root — NOT inside directories as `index.html`. This prevents Cloudflare Pages trailing-slash canonical mismatches.
